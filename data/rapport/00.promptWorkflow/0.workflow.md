# Workflow — Implémentation par lots avec Codex

Ce workflow décrit la méthode utilisée pour implémenter des lots techniques de manière progressive, maîtrisée et sans dérive.

Objectifs :
- cadrer Codex pour éviter les implémentations approximatives
- garder le contrôle architectural
- itérer proprement avec analyse → implémentation → correction

---

## 1. Analyse globale du lot

### 1.1 Prompt d’analyse initial

- Rédiger un prompt demandant à Codex :
    - d’analyser l’existant
    - d’identifier les problèmes
    - de proposer une approche globale

- Codex génère un rapport : data/rapport/vX.X/lotX/result_global_analysis.md

---

## 2. Élaboration de la roadmap

À partir du rapport global :

- découper le lot en **phases cohérentes**
- ordonner les phases de manière progressive
- éviter les dépendances circulaires

Chaque phase doit être :
- indépendante
- testable
- à faible risque

---

## 3. Boucle par phase

Pour chaque phase, appliquer le cycle suivant :

---

### 3.1 Analyse de la phase

#### Objectif
Comprendre précisément :
- ce qui doit être modifié
- comment l’intégrer dans l’existant

#### Action
- rédiger un prompt d’analyse ciblé pour Codex

#### Résultat attendu
data/rapport/vX.X/lotX/result_[numPhase]_analysis.md


---

### 3.2 Cadrage de l’implémentation

#### Objectif
- éviter toute dérive de Codex
- imposer un périmètre strict

#### Action
- analyser le rapport Codex
- corriger les mauvaises orientations si besoin
- rédiger un prompt d’implémentation détaillé :
    - contraintes strictes
    - décisions validées
    - structure attendue
    - limites à ne pas dépasser

---

### 3.3 Implémentation par Codex

Codex exécute le prompt d’implémentation et produit :
data/rapport/vX.X/lotX/result_[numPhase]_implements.md


+ une relecture critique de son propre travail

---

### 3.4 Audit et corrections

#### Objectif
- détecter :
    - dérives
    - mauvaises pratiques
    - dette technique introduite

#### Action
- analyser :
    - rapport d’implémentation
    - relecture Codex

- rédiger un prompt de correction ciblé

#### Résultat attendu
data/rapport/vX.X/lotX/result_[numPhase]_correction.md


---

### 3.5 Boucle de validation

- si nécessaire → nouvelle boucle correction
- sinon → phase validée

---

## 4. Règles importantes du workflow

### 4.1 Toujours analyser avant d’implémenter
❌ Ne jamais sauter l’étape d’analyse  
✔️ Même pour des tâches simples

---

### 4.2 Ne jamais laisser Codex libre

Toujours fournir :
- contraintes explicites
- périmètre clair
- décisions déjà prises

---

### 4.3 Favoriser le minimalisme

- implémentation simple
- pas de sur-architecture
- pas d’anticipation excessive

---

### 4.4 Corriger immédiatement la dette structurelle

- couplage de couches
- config mal placée
- duplication
- magic strings

👉 corriger pendant la phase, pas après

---

### 4.5 Une seule source de vérité

- éviter les configs parallèles
- éviter les systèmes concurrents
- toujours réutiliser l’existant si pertinent

---

### 4.6 Prioriser la valeur d’exploitation

Ordre typique :
1. données critiques métier
2. supervision
3. diagnostic
4. optimisation

---

## 5. Clôture du lot

### 5.1 Phase finale — Bilan global

Créer :
data/rapport/vX.X/lotX/z_finalRepport.md


Contenu :
- ce qui a été implémenté
- ce qui n’a pas été fait
- dettes techniques assumées
- décisions structurantes
- impacts
- recommandations futures

---

### 5.2 Objectif du rapport final

- servir de base pour les prochains lots
- documenter les choix d’architecture
- éviter de “réinventer” les décisions

---

## 6. Résumé du cycle

```text
Analyse globale
→ Roadmap
→ (par phase)
    Analyse
    → Implémentation cadrée
    → Exécution Codex
    → Audit
    → Corrections
    → Validation
→ Rapport final
```

---

## 7. Bénéfices du workflow
- maîtrise complète des implémentations Codex
- réduction drastique de la dette technique
- architecture cohérente dans le temps
- facilité de reprise dans de nouvelles discussions
