Tu interviens sur mon projet Java Spring “Toolkit-Bridge”, une plateforme agentique générique en cours de construction.

Objectif global :
construire une architecture agentique multi-agent, modulaire, maintenable, extensible et propre, sans spaghetti, avec une montée en complexité progressive et organisée.

Contraintes d’architecture à respecter impérativement :
- Toujours privilégier une architecture claire, découplée et extensible.
- Ne pas introduire de logique ad hoc ou de couplage caché entre sous-systèmes.
- Ne pas casser l’existant sans nécessité forte.
- Préférer des contrats explicites (interfaces, objets métier, types) plutôt que des échanges implicites.
- Toute nouvelle brique doit préparer les futures évolutions sans sur-ingénierie inutile.
- Le code doit rester lisible, testable et cohérent avec une progression incrémentale.
- Les noms de classes doivent être en anglais.
- Rester compatible avec l’esprit actuel du projet : Spring Boot, architecture propre, orientation multi-agent, usage futur de mémoire, tasks, outils, communication inter-agent et supervision.

Méthode de travail attendue :
1. Lire le code existant et comprendre l’architecture actuelle avant de modifier.
2. Identifier précisément les classes/fichiers concernés.
3. Implémenter uniquement le périmètre demandé.
4. Si une petite adaptation annexe est nécessaire pour garder une architecture saine, tu peux la faire, mais sans étendre inutilement le scope.
5. Produire une implémentation propre, cohérente et prête à servir de base aux phases suivantes.
6. Ajouter ou adapter les tests pertinents.
7. Ajouter du logging utile seulement si cela aide réellement à l’exploitation ou au debug.
8. Documenter brièvement les choix structurants directement dans le code si nécessaire, ou dans un petit fichier markdown si cela clarifie l’architecture.

Exigences de qualité :
- Pas de rustine rapide.
- Pas de duplication évitable.
- Pas de magie cachée.
- Pas de logique métier dispersée.
- Pas de dépendance forte entre composants si une abstraction simple peut l’éviter.
- Les responsabilités doivent être claires.
- Les objets métier doivent être introduits tôt si cela évite une future refactorisation lourde.

Livrables attendus à chaque phase :
- code modifié
- tests ajoutés ou ajustés
- bref résumé de ce qui a été fait
- points d’attention éventuels
- liste claire des fichiers créés/modifiés
