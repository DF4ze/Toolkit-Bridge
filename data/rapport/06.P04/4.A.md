Tu vas implémenter la phase 4.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le runtime multi-agent, les orchestrators et le modèle de tâche.

Objectif de cette phase :
mettre en place un bus interne de communication inter-agent avec une abstraction de message explicite, typée, simple et extensible.

Cette phase ne doit pas encore implémenter :
- une résolution avancée par rôle
- un load balancing
- une délégation complexe multi-agent
- une persistance complète du bus
- une supervision Telegram branchée réellement
- un système distribué ou asynchrone complexe

Le but est de poser un bus interne propre avec de vrais messages typés.

Périmètre attendu :

1. Mettre en place un bus inter-agent
   Je veux une abstraction claire de bus de communication interne entre agents.
   Ce bus doit devenir le point de passage propre pour les échanges inter-agent.

Attention :
- ne pas créer un composant trop “intelligent”
- ne pas y mettre prématurément toute la logique métier
- garder un design simple, lisible et testable

2. Créer une abstraction de message
   Je veux une enveloppe de message typée contenant au minimum :
- un identifiant de message
- un identifiant de corrélation
- un émetteur
- un destinataire
- un horodatage
- un type de message
- un payload ou contenu structurable si pertinent

Cette abstraction doit préparer :
- la supervision
- la traçabilité
- une future persistance

Sans implémenter maintenant toute la mécanique autour.

3. Définir les types de messages V1
   Je veux au minimum les types suivants :
- `TASK_REQUEST`
- `TASK_RESPONSE`
- `QUESTION`
- `ANSWER`
- `EVENT`
- `CRITIQUE`

L’objectif est de rester minimal mais explicite, et d’éviter le tout-texte non cadré.

4. Garder le bus indépendant des futures stratégies de résolution
   Cette phase ne doit pas encore figer la logique complète de ciblage par rôle.
   Le bus doit rester compatible avec :
- un destinataire agent
- un futur destinataire par rôle

Mais sans embarquer déjà toute la stratégie de résolution avancée.

5. Préparer supervision et persistance sans les implémenter complètement
   Le modèle de message et le bus doivent être pensés de façon compatible avec :
- des traces ou événements de supervision
- une future persistance des messages importants

Mais sans sur-implémentation.

Attendus techniques :
- créer ou adapter les classes/interfaces nécessaires
- garder les responsabilités nettes entre bus, message et logique métier
- rester cohérent avec les phases précédentes
- éviter les dépendances inutiles
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- bus fourre-tout
- messages trop flous ou purement textuels
- couplage fort avec Telegram
- couplage fort avec le futur système de résolution par rôle
- sur-ingénierie
- simulation prématurée d’un système distribué

Livrables attendus :
1. code complet de la phase 4.A
2. tests
3. résumé court des choix autour du bus et des messages
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 4.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
