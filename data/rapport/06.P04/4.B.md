Tu vas implémenter la phase 4.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 4.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le bus inter-agent et le runtime multi-agent déjà en place.

Objectif de cette phase :
préparer l’adressage par rôle en découplant rôle et instance d’agent, avec une résolution simple, interchangeable et maintenable.

Cette phase ne doit pas encore implémenter :
- un load balancing avancé
- des stratégies complexes de scheduling
- un système distribué
- une supervision complète du routage
- une orchestration multi-agent avancée

Le but est de permettre un ciblage par rôle sans figer trop tôt une logique compliquée.

Périmètre attendu :

1. Préparer l’adressage par rôle
   Je veux qu’un message ou une demande puisse viser :
- soit un agent précis
- soit un rôle

Le système doit être conçu pour permettre la délégation à un rôle plutôt qu’à une instance nominative.

2. Découpler rôle et instance
   Je veux éviter qu’un rôle soit lié rigidement à un agent unique.
   L’architecture doit permettre plus tard :
- remplacement transparent d’un agent
- plusieurs agents pour un même rôle
- futur équilibrage de charge

Sans implémenter maintenant tout ce comportement.

3. Prévoir une résolution simple V1
   Je veux une première stratégie de résolution simple :
- sélectionner d’abord un agent libre correspondant au rôle si c’est pertinent
- ou une stratégie minimale équivalente si l’état runtime actuel le permet mieux

Attention :
- cette stratégie doit rester interchangeable
- elle ne doit pas être figée dans le bus lui-même
- elle ne doit pas devenir une logique omnisciente

4. Garder une séparation claire des responsabilités
   Je veux éviter que :
- le bus décide de toute la logique métier
- le runtime registry absorbe tout le routage
- la résolution soit dispersée un peu partout

Il faut un point clair, simple et testable pour cette responsabilité.

5. Préparer les futures évolutions
   Cette phase doit préparer plus tard :
- load balancing
- sélection plus fine selon disponibilité
- supervision des agents occupés/libres
- routage plus intelligent

Mais sans les implémenter prématurément.

Attendus techniques :
- créer ou adapter les classes/interfaces nécessaires
- brancher proprement la résolution sur le runtime multi-agent existant
- garder les responsabilités nettes
- rendre la stratégie de résolution interchangeable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- logique de résolution codée en dur dans le bus
- dépendance forte à une unique stratégie
- load balancing prématuré
- routage trop complexe en V1
- couplage fort avec des phases futures non encore implémentées

Livrables attendus :
1. code complet de la phase 4.B
2. tests
3. résumé court des choix autour de l’adressage par rôle et de la résolution
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
