Après relecture :
Revue Critique (findings)

[P1] Compatibilité legacy partiellement cassée sur les scopes: SHARED pouvait être visible dans la policy mais jamais récupéré car absent des scopes de requête par défaut (MemoryScopePolicy.java:34, DefaultMemoryRetrievalFacade.java:65).
[P1] Trimming du contexte fragile: risque de couper l’input utilisateur ou de dégrader l’ordre utile en fin d’assemblage (DefaultContextAssembler.java:35).
[P2] Sémantique legacy discutable: SHARED était remappé vers USER dans l’assemblage, ce qui n’est pas neutre métier (DefaultContextAssembler.java:139).
[P3] Couverture de tests incomplète sur les points introduits (priorité de trimming et normalisation userId) (DefaultContextAssemblerTest.java:195, ContextRequestTest.java:23).
Corrections Proposées

Inclure explicitement SHARED dans les scopes lisibles par défaut.
Rendre le trimming priorisé: conserver User Input, puis injecter rules/facts/history/conversation selon budget.
Remapper SHARED vers SYSTEM (plus neutre que USER).
Ajouter les tests ciblés sur ces comportements.
Corrections Appliquées

SHARED ajouté aux scopes de lecture par défaut (MemoryScopePolicy.java:34).
Assemblage contextuel refactoré avec budget priorisé (assembleWithPriority) et conservation de User Input (DefaultContextAssembler.java:192).
Remap legacy SHARED -> SYSTEM côté rendu contexte (DefaultContextAssembler.java:139).
Tests renforcés:
trimming + conservation user input (DefaultContextAssemblerTest.java:87, DefaultContextAssemblerTest.java:195)
scopes retrieval incluant SHARED (DefaultMemoryRetrievalFacadeTest.java:231)
normalisation userId dans ContextRequest (ContextRequestTest.java:12).

Résumé Final
La séparation config/runtime reste propre, le découplage orchestrator/memory est préservé, et la phase est plus robuste pour la suite (permissions/retrieval plus fin) sans ajouter de fonctionnalité hors périmètre.
