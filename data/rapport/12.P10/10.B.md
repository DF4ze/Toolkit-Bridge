Tu vas implémenter la phase 10.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 10.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur la policy par agent déjà introduite.

Objectif de cette phase :
centraliser le contrôle des permissions afin que toute action sensible passe par un point de vérification clair, testable et durable.

Cette phase ne doit pas encore implémenter :
- une validation humaine réelle
- une politique contextuelle complexe
- un système de sécurité distribué
- des workflows d’approbation avancés
- un audit complet multi-niveaux

Le but est de s’assurer qu’aucune décision sensible ne soit disséminée dans le code.

Périmètre attendu :

1. Centraliser le contrôle
   Je veux qu’il existe un point clair de vérification des permissions avant exécution des actions sensibles.
   Le système doit éviter que les décisions de sécurité soient dispersées dans les orchestrators, tools ou services métier.

2. Vérifier avant exécution
   Je veux que toute action sensible passe par la policy avant exécution.
   Cela concerne au minimum les capacités définies dans la phase 10.A, selon ce qui est pertinent dans l’état actuel du projet.

3. Journaliser les refus
   Je veux que les refus ou interdictions soient journalisés de manière utile.
   Attention :
- logging lisible
- pas de bruit inutile
- cohérence avec l’observabilité existante

4. Préparer la montée en finesse
   L’architecture mise en place ici doit être compatible plus tard avec :
- validation humaine
- règles plus fines
- décisions plus contextuelles

Mais sans les implémenter maintenant.

5. Garder le contrôle simple et maintenable
   Je veux éviter :
- un composant omniscient trop complexe
- des décisions de sécurité cachées
- une logique difficile à faire évoluer
- une dépendance trop forte à des phases futures non encore faites

Attendus techniques :
- créer ou adapter les classes/interfaces/services nécessaires
- brancher proprement les contrôles sur les points d’exécution sensibles existants
- garder les responsabilités nettes entre policy, contrôle et exécution
- rendre le système lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- vérifications dispersées
- contrôle codé directement dans chaque tool ou orchestrator
- journalisation incohérente
- architecture trop lourde pour une V1
- anticipation excessive d’une validation humaine non encore implémentée

Livrables attendus :
1. code complet de la phase 10.B
2. tests
3. résumé court des choix autour du contrôle centralisé des permissions
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
