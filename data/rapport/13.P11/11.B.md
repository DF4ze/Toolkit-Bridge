Tu vas implémenter la phase 11.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 11.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur la catégorisation des tools déjà introduite, ainsi que sur le système de tools existant et la policy par agent.

Objectif de cette phase :
structurer un registre de tools homogène, descriptif et exploitable, afin de mieux décrire les outils disponibles, supporter le filtrage par agent et préparer les futures extensions de la plateforme.

Cette phase ne doit pas encore implémenter :
- un marketplace interne de tools
- une UI d’administration complète
- un système complet de scripted tools persistants
- une validation multi-agent
- une gouvernance avancée du cycle de vie des tools

Le but est de poser un registre propre et durable.

Périmètre attendu :

1. Structurer le registre de tools
   Je veux que la plateforme dispose d’un registre de tools homogène et explicite.
   Ce registre doit permettre de décrire les tools au-delà de leur simple exécution technique.

2. Décrire chaque tool proprement
   Je veux qu’un tool puisse être décrit au minimum par des informations du type :
- nom
- catégorie
- description
- schéma
- capacités
- niveau de risque

Attention :
- rester cohérent avec les contrats déjà présents dans le projet
- ne pas multiplier les métadonnées purement décoratives
- garder un modèle utile, lisible et exploitable

3. Supporter le filtrage par agent
   Je veux que le registre puisse préparer ou supporter le filtrage des tools selon l’agent, en lien avec les policies.
   Attention :
- le registre ne doit pas devenir lui-même la policy
- il doit faciliter l’exposition contrôlée des tools

4. Mieux exposer les possibilités aux LLM
   Le registre doit aider à fournir une vision homogène des tools disponibles aux orchestrators et aux LLM.
   Je veux éviter les implémentations ad hoc ou la construction dispersée de la description des tools.

5. Préparer les futures extensions
   Cette phase doit préparer proprement des familles de tools comme :
- web
- git
- mémoire
- fichiers
- exécution
- agents
- scripts

Mais sans imposer dès maintenant une architecture trop lourde ou trop théorique.

6. Garder une séparation claire des responsabilités
   Je veux éviter de confondre :
- registre de tools
- exécution des tools
- policy
- validation future

Le registre doit décrire et exposer proprement, sans absorber toute la logique de contrôle.

Attendus techniques :
- créer ou adapter les classes/interfaces/services nécessaires
- rendre le registre homogène et maintenable
- rester cohérent avec les catégories, niveaux de risque, policies et tools existants
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- registre décoratif ou inutilisé
- doublons entre registre et implémentation des tools
- filtre d’agent codé en dur dans plusieurs endroits
- architecture trop théorique
- sur-conception

Livrables attendus :
1. code complet de la phase 11.B
2. tests
3. résumé court des choix autour du registre de tools
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
