Tu vas implémenter la phase 12.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment la plateforme de tools, le registre de tools, les policies par agent, le système de workspace et la distinction `native` / `scripted` déjà introduite.

Objectif de cette phase :
poser le support technique minimal des scripted tools, avec un stockage persistant, des métadonnées sauvegardées, et une activation explicite, sans faire entrer toute la V2 dans le chemin critique.

Cette phase ne doit pas encore implémenter :
- un moteur complet d’exécution de scripted tools
- une UI d’administration complète
- une validation humaine réelle
- un workflow complet de revue
- une gouvernance avancée du cycle de vie
- un marketplace interne de tools

Le but est de préparer proprement la V2, sans bloquer la V1 ni lancer un gros chantier prématuré.

Périmètre attendu :

1. Poser l’emplacement des scripted tools
   Je veux qu’il existe une base claire et persistante pour les scripted tools.
   Un tool développé doit survivre au redémarrage.

2. Prévoir un stockage persistant
   Je veux distinguer proprement :
- les métadonnées du scripted tool
- le contenu/script lui-même

Les métadonnées doivent pouvoir être sauvegardées et relues.
Le contenu doit être persisté lui aussi.

3. Prévoir des métadonnées versionnées
   Je veux que les scripted tools aient des métadonnées suffisamment propres pour préparer plus tard :
- version
- état
- activation
- origine éventuelle

Attention :
- rester simple en V1 préparatoire
- ne pas construire un système de versioning lourd

4. Garder une activation explicite
   Je veux qu’un scripted tool ne soit pas considéré comme automatiquement utilisable juste parce qu’il existe en stockage.
   Il faut préparer une notion d’activation explicite.

5. Garder V2 hors du chemin critique
   Cette phase ne doit pas faire dépendre la V1 d’un système complet de scripted tools.
   Je veux seulement les abstractions et le support minimal nécessaires pour ne pas créer de dette technique.

6. Rester cohérent avec la plateforme de tools existante
   Je veux éviter :
- un sous-système totalement à part
- une rupture avec le registre de tools
- une confusion entre scripted tool stocké et scripted tool effectivement exposé

Attendus techniques :
- créer ou adapter les classes/interfaces/services/properties nécessaires
- garder les responsabilités nettes entre stockage, métadonnées, activation et exposition future
- rester cohérent avec le registre de tools et les policies
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- implémentation prématurée d’un système complet de scripted tools
- stockage flou ou non durable
- activation implicite
- abstraction trop lourde
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 12.A
2. tests
3. résumé court des choix autour du support persistant des scripted tools
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 12.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
