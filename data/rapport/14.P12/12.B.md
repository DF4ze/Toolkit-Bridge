Tu vas implémenter la phase 12.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 12.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur la base de scripted tools déjà introduite, ainsi que sur la plateforme de tools, les policies et les niveaux de risque existants.

Objectif de cette phase :
préparer la validation future des scripted tools en introduisant les bonnes abstractions autour des classes de risque, des niveaux de validation et de la future revue multi-agent, sans implémenter le workflow complet.

Cette phase ne doit pas encore implémenter :
- une validation humaine réelle en production
- un moteur complet de revue multi-agent
- une approbation automatisée complexe
- une orchestration complète sécurité/dev/finance/etc.
- une UI de validation

Le but est de poser une architecture de gouvernance future, pas le système final.

Périmètre attendu :

1. Anticiper plusieurs classes de risque
   Je veux que l’architecture prépare explicitement plusieurs classes de risque pertinentes pour les scripted tools, par exemple :
- lecture seule
- écriture
- exfiltration
- modification système

L’objectif est de permettre plus tard des niveaux de validation différents selon le risque.

2. Préparer un support pour validation future
   Je veux préparer la possibilité qu’un scripted tool soit soumis plus tard à :
- validation humaine
- validation automatisée
- ou autre workflow d’approbation

Attention :
- ne pas implémenter ici le workflow complet
- poser seulement les abstractions et états nécessaires

3. Ouvrir la voie à la revue multi-agent
   Je veux préparer l’idée suivante :
- un agent dev produit un tool
- un agent sécurité ou autre rôle peut l’analyser
- un humain tranche sur les cas sensibles

Cette phase doit ouvrir la voie à ce scénario, sans le construire entièrement.

4. Garder une séparation claire des responsabilités
   Je veux éviter que :
- le stockage décide de la validation
- la policy absorbe toute la gouvernance de scripted tools
- le registre de tools devienne le moteur complet d’approbation

Les responsabilités doivent rester propres et extensibles.

5. Ne pas figer trop tôt le workflow final
   Je veux éviter une implémentation prématurée du workflow réel.
   Cette phase doit rester préparatoire et structurante.

Attendus techniques :
- créer ou adapter les classes/interfaces/enums/services nécessaires
- rendre visibles les notions de classe de risque, état de validation ou abstraction équivalente si pertinent
- rester cohérent avec les phases précédentes
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- workflow complet prématuré
- validation codée en dur
- confusion entre risque, policy et approbation
- architecture trop théorique
- sur-conception

Livrables attendus :
1. code complet de la phase 12.B
2. tests
3. résumé court des choix autour de la validation future et de la revue multi-agent des scripted tools
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
