Tu vas implémenter la phase 13.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment les runtimes agents, les tasks, le bus inter-agent, les tools, les policies et les orchestrators.

Objectif de cette phase :
séparer proprement les logs techniques des traces agentiques métier, et introduire une base de traces structurées au format JSON pour faciliter debug, audit et analyse.

Cette phase ne doit pas encore implémenter :
- un dashboard complet
- une UI d’administration
- une observabilité distribuée complexe
- une stack complète de monitoring externe
- une méta-analyse avancée
- un catalogue exhaustif d’événements métier

Le but est de poser une base d’observabilité claire, structurée et durable.

Périmètre attendu :

1. Conserver les logs applicatifs classiques
   Je veux garder les logs techniques existants pour :
- erreurs Java
- stacktraces
- infra
- Spring

L’objectif est de rester compatible avec l’écosystème standard et de ne pas surcharger les logs existants avec des données métier mal adaptées.

2. Créer des traces agentiques structurées
   Je veux introduire des traces métier distinctes des logs techniques.
   Ces traces doivent être structurées, idéalement en JSON, et exploitables pour :
- debug
- audit
- analyse future

3. Poser la corrélation des traces
   Je veux que les traces agentiques soient corrélables avec des identifiants explicites, au minimum selon ce qui est pertinent dans l’état actuel du projet :
- `runId`
- `taskId`
- `agentId`
- `messageId`

Attention :
- ne pas inventer des identifiants inutiles si certains n’existent pas encore partout
- mais préparer clairement la corrélation inter-briques

4. Séparer clairement logs techniques et traces métier
   Je veux éviter :
- que les logs applicatifs deviennent un pseudo stockage métier
- que les traces agentiques soient noyées dans les logs techniques
- que les producteurs d’événements aient à gérer eux-mêmes tout le format de sortie

5. Garder le système simple et extensible
   Je veux une base saine, pas une usine d’observabilité.
   Cette phase doit préparer plus tard :
- catalogue d’événements enrichi
- destinations multiples
- UI d’admin
- supervision Telegram
- méta-analyse

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services nécessaires
- introduire une abstraction propre pour les traces agentiques
- garder les responsabilités nettes entre logs techniques, traces métier et producteurs d’événements
- rendre le système lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- confusion entre logs techniques et traces métier
- JSON bricolé à la main partout dans le code
- corrélation dispersée et incohérente
- stockage métier reconstruit depuis les logs
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 13.A
2. tests
3. résumé court des choix autour des traces agentiques structurées
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 13.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
