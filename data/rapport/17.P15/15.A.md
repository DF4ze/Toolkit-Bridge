Tu vas implémenter la phase 15.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le runtime multi-agent, le bus inter-agent, le modèle de tâche, le modèle d’artefact, les traces agentiques et la supervision existante.

Objectif de cette phase :
définir le débat inter-agent comme un protocole explicite, traçable et encadré, afin d’introduire une contradiction utile entre rôles complémentaires sans créer un système multi-agent incontrôlable.

Cette phase ne doit pas encore implémenter :
- un grand système de débat généralisé entre de nombreux agents
- un vote complexe
- un arbitrage avancé
- une orchestration autonome très riche
- une UI dédiée au débat
- une logique métier dépendante d’un type de débat unique

Le but est de poser un protocole simple, propre et réutilisable.

Périmètre attendu :

1. Formaliser les rôles complémentaires
   Je veux préparer explicitement des rôles complémentaires dans le cadre d’un débat inter-agent, par exemple :
- planner
- critic
- sécurité
- finance
- commercial
- exécuteur

Attention :
- ne pas figer un catalogue énorme ou définitif
- garder le modèle ouvert et cohérent avec les rôles déjà introduits dans le runtime
- chaque rôle doit représenter une lecture différente et utile

2. Définir le débat comme un protocole
   Je veux un protocole explicite d’échange inter-agent, avec au minimum une structure de type :
- question initiale
- réponses
- critiques
- synthèse

Le système doit éviter le simple échange textuel non cadré.

3. Encadrer les échanges
   Je veux une corrélation stricte des messages du débat.
   Il faut pouvoir relier clairement :
- l’initiation du débat
- les réponses associées
- les critiques
- la synthèse finale

L’objectif est aussi de limiter la dérive conversationnelle et le bavardage inutile.

4. Rester cohérent avec le bus existant
   Je veux que le protocole de débat s’appuie proprement sur le bus inter-agent et les abstractions déjà présentes.
   Je veux éviter :
- un sous-système parallèle totalement séparé
- des messages de débat codés de manière ad hoc
- une logique de corrélation bricolée

5. Garder la mécanique petite et extensible
   Cette phase doit préparer plus tard :
- davantage de rôles
- arbitrage
- synthèse plus riche
- protocoles spécialisés

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/enums/services/types nécessaires
- rendre le protocole explicite, lisible et testable
- rester cohérent avec le bus, les tâches, les artefacts et les traces
- garder les responsabilités nettes
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- grand système de débat prématuré
- échange purement textuel sans cadre
- corrélation floue
- protocole trop spécifique à un seul cas d’usage
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 15.A
2. tests
3. résumé court des choix autour du protocole de débat inter-agent
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 15.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
