Tu vas implémenter la phase 17.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment les tasks, messages, traces, artefacts, mémoires, scripted tools et mécanismes de stockage déjà introduits.

Objectif de cette phase :
typer explicitement les objets persistables du système, afin de clarifier ce qui est durable, ce qui est éphémère, et d’éviter toute persistance implicite ou ambiguë.

Cette phase ne doit pas encore implémenter :
- un système complet d’archivage
- une purge automatique complexe
- une stratégie de stockage distribuée
- une UI d’administration dédiée à la rétention
- une politique réglementaire avancée de conservation

Le but est de poser un modèle clair, compréhensible et maintenable autour de la persistance.

Périmètre attendu :

1. Définir les familles durables
   Je veux typer explicitement les grandes familles d’objets persistables du système, au minimum selon ce qui existe réellement dans le projet :
- tasks
- messages
- traces
- artefacts
- mémoires
- scripted tools

Chaque type doit pouvoir être compris comme une famille durable distincte avec son propre cycle de vie.

2. Clarifier éphémère vs durable
   Je veux que l’architecture rende explicite la différence entre :
- objets temporaires / éphémères
- objets destinés à être persistés

L’objectif est d’éviter les ambiguïtés et les persistances accidentelles.

3. Éviter les objets hybrides
   Je veux éviter les objets “mi-runtime mi-stockage” ou “mi-temporaire mi-durable”.
   Si un objet a un cycle de vie particulier, cela doit être visible et compréhensible.

4. Ne pas laisser la persistance implicite
   Je veux éviter que la persistance soit décidée de façon opportuniste au détour d’un service ou d’un handler.
   Les choix de durabilité doivent être explicites dans le modèle ou dans l’infrastructure adaptée.

5. Faciliter maintenance et nettoyage
   Le typage introduit ici doit préparer plus tard :
- règles de rétention
- purge
- archivage
- administration

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/enums/modèles nécessaires
- rendre visible la notion de famille persistable ou abstraction équivalente si pertinent
- rester cohérent avec les briques déjà en place
- garder le code lisible, testable et maintenable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- persistance implicite disséminée
- familles durables floues
- objets hybrides mal définis
- abstraction trop lourde ou trop théorique
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 17.A
2. tests
3. résumé court des choix autour du typage des objets persistables
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 17.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
