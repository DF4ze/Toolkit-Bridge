Tu vas implémenter la phase 18.2.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 18.1.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur la cartographie/matrice de migration déjà produite.

Objectif de cette phase :
définir et mettre en place la nouvelle source de vérité de la configuration administrable, puis refactorer les accès pour sortir progressivement des lectures directes du YAML devenues inadaptées.

Cette phase ne doit pas encore implémenter :
- la UI d’administration
- des endpoints REST d’admin complets
- le bootstrap complet de first-initialisation
- du RW applicatif sur `application.yml`
- une migration totale de tous les paramètres si certains ne sont pas encore mûrs

Le but est de poser le bon modèle de sources de vérité et de refactorer proprement les accès structurants.

Périmètre attendu :

1. Définir la nouvelle source de vérité
   Je veux rendre explicite la stratégie suivante :
- le YAML reste la source de bootstrap et de configuration technique
- la base de données devient la source de vérité pour les paramètres administrables modifiables à chaud
- les fichiers dédiés restent la cible pour les contenus métier lisibles/éditables

Cette frontière doit devenir visible dans l’architecture.

2. Refactorer l’accès à la configuration administrable
   Je veux introduire des services, repositories ou abstractions dédiées à la lecture/écriture de configuration administrable.
   L’objectif est de remplacer progressivement les lectures directes du YAML devenues inadaptées.

Attention :
- ne pas disperser la logique dans plusieurs composants
- ne pas mélanger configuration technique et configuration administrable
- rester cohérent avec Spring Boot et l’architecture existante

3. Remplacer les accès directs les plus critiques
   En s’appuyant sur la cartographie précédente, je veux refactorer les zones les plus structurantes qui lisent directement une config devenue métier.
   Le refactoring doit être :
- ciblé
- propre
- testable
- incrémental

4. Préparer l’évolution sans dépendre du YAML en écriture
   Je veux explicitement éviter que `application.yml` devienne la base de vérité vivante de l’administration.
   Le système doit préparer la persistance durable des changements futurs via la base ou un stockage approprié.

5. Garder la migration progressive
   Je ne veux pas une énorme refonte incontrôlée.
   Si tous les paramètres ne peuvent pas être migrés proprement dans cette phase, priorise les plus structurants et laisse les autres explicitement listés pour la suite.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/repositories/entities nécessaires
- rendre la séparation YAML / DB / fichiers visible et compréhensible
- refactorer les accès les plus importants identifiés en phase 18.1.A
- garder le code lisible, maintenable et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- RW sur `application.yml` comme mécanisme principal
- lectures de config métier encore disséminées partout
- migration trop brutale et risquée
- confusion entre config technique et config administrable
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 18.2.A
2. tests
3. résumé court des choix autour de la nouvelle source de vérité
