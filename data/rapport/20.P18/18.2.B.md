Tu vas implémenter la phase 18.2.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases 18.1.A et 18.2.A déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur la nouvelle stratégie de source de vérité déjà introduite.

Objectif de cette phase :
mettre en place un bootstrap de first-initialisation permettant d’initialiser la configuration administrable en base lorsqu’elle n’existe pas encore, en utilisant le YAML comme seed initial et non comme source vivante principale.

Cette phase ne doit pas encore implémenter :
- une UI d’administration
- une édition directe du YAML
- un mécanisme de resynchronisation complexe YAML ↔ DB
- une migration automatique infiniment fine de tous les cas limites

Le but est de garantir un démarrage cohérent sans reconfiguration manuelle complète, tout en gardant une architecture saine.

Périmètre attendu :

1. Mettre en place un bootstrap de first-initialisation
   Je veux qu’au démarrage, si aucune configuration administrable n’existe en base, le système puisse l’initialiser avec des valeurs par défaut cohérentes.

2. Utiliser le YAML comme seed initial
   Le YAML doit être utilisé ici comme source d’initialisation par défaut, pas comme source de vérité vivante principale.
   Il faut que cette responsabilité soit claire et limitée.

3. Garantir un démarrage cohérent
   Je veux éviter qu’un redémarrage de l’application oblige à tout reconfigurer manuellement.
   Le système doit être capable de démarrer proprement avec une configuration persistée.

4. Rester sain côté architecture
   Je veux éviter :
- tout RW sur `application.yml` comme mécanisme principal
- toute confusion entre seed initial et configuration courante
- une logique de bootstrap disséminée dans plusieurs composants

5. Préparer les futures interfaces d’administration
   Cette phase doit préparer proprement :
- endpoints REST d’administration
- future UI web d’administration
- persistance durable des changements faits depuis l’admin

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/initializers nécessaires
- brancher proprement le bootstrap sur la nouvelle couche de configuration administrable
- rendre explicite la différence entre config bootstrap et config courante
- garder le code lisible, maintenable et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- logique de bootstrap implicite
- RW dans le YAML
- seed et source de vérité confondues
- gros mécanisme de migration prématuré
- sur-conception

Livrables attendus :
1. code complet de la phase 18.2.B
2. tests
3. résumé court des choix autour du bootstrap de first-initialisation
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour la future phase 18.3
