# Refacto des endpoints d’administration web pour Marcel / Tool-Bridge

Tu es un développeur senior Java Spring Boot.
Tu travailles sur mon projet **Tool-Bridge**, future UI d’administration **Marcel**.

## Contexte

Je vais mettre en place une interface d’administration web en :
- Spring MVC
- Thymeleaf
- Bootstrap
- jQuery

Avant de construire les pages, je veux **refactorer les endpoints d’administration** pour avoir une base propre, cohérente, REST-ish, exploitable facilement par une future UI serveur-side.

Aujourd’hui, il existe déjà plusieurs controllers REST côté web, mais ils sont hétérogènes :
- auth courant
- création d’agent
- debug LLM
- administration technique

Je veux une structure plus propre, plus cohérente, plus facile à maintenir.

---

## Objectif global

Refactorer les endpoints web d’administration pour obtenir :

1. une séparation claire entre :
    - **auth**
    - **admin technique / observabilité**
    - **admin fonctionnelle / configuration**

2. des routes cohérentes

3. la suppression des endpoints ambigus ou peu REST, notamment les créations via `GET`

4. une base saine pour la future UI MVC Thymeleaf

---

## Ce que je veux comme résultat

Je veux que tu :
- analyses l’existant
- proposes puis implémentes un refacto propre
- minimises les régressions
- gardes le code lisible

Je veux aussi un petit résumé final :
- anciens endpoints

## Principes d’architecture à respecter

### 1. Séparer les controllers par responsabilité

Je veux quelque chose de ce genre :

- `...controller.web.auth`
- `...controller.web.admin.technical`
- `...controller.web.admin.config` ou `...controller.web.admin.functional`

Adapte au package existant du projet, mais garde une séparation claire.

### 2. Conserver `/api/...` pour les endpoints REST

Je veux conserver une convention claire du type :

- `/api/auth/...`
- `/api/admin/technical/...`
- `/api/admin/llms/...`
- `/api/admin/telegram-bots/...`
- `/api/admin/agents/...`

### 3. Pas de création via GET

Je veux supprimer les endpoints du style :
- `GET /.../new` qui créent une ressource
- ou tout autre endpoint GET ayant un effet de bord

Les créations doivent être en `POST`.

### 4. Base saine pour future UI

Je veux des endpoints qui serviront bien :
- soit à une UI MVC Thymeleaf
- soit éventuellement à du JS léger

Donc :
- routes lisibles
- réponses lisibles
- paramètres cohérents
- pas de conventions bizarres

# B. Agents (administration fonctionnelle)

### Problème actuel
Il existe actuellement une logique de création d’agent avec des routes peu propres, notamment avec création via GET.

### Attendu
Je veux un vrai controller REST admin pour les agents.

Créer ou refactorer vers des endpoints de ce type :

- `GET /api/admin/agents`
    - liste des agents
- `GET /api/admin/agents/{agentId}`
    - détail d’un agent
- `POST /api/admin/agents`
    - création d’un agent
- éventuellement plus tard :
    - `PUT /api/admin/agents/{agentId}`
    - `DELETE /api/admin/agents/{agentId}`

### Important
Même si tout n’est pas encore implémentable côté service, je veux au minimum :
- une structure propre
- la suppression du `GET` créateur
- des noms cohérents

### Si l’existant ne permet pas encore la liste/détail
Alors :
- mets en place la structure controller + DTO + signatures propres
- fais au mieux avec les services existants
- laisse un code proprement extensible

### Attendu

### Proposition cible

- `GET /api/admin/llms`

Tu peux choisir la convention la plus cohérente avec l’existant, mais elle doit être :
- explicite
- propre
- justifiable

### Important
Le debug LLM doit être clairement identifié comme du **technique/debug**, pas comme du CRUD métier.

Si tu juges plus sain de le déplacer côté `technical`, fais-le.
