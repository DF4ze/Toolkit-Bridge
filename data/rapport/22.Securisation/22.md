# Implémentation d’une sécurité web simple pour l’admin Marcel

Tu es un développeur senior Java Spring Boot.

Tu travailles sur mon projet **Tool-Bridge**, future interface d’administration **Marcel**.

## Contexte

Je veux mettre en place une sécurité web **simple, légère, efficace**, sans système complet d’utilisateurs/mots de passe.

L’objectif est :
- de bloquer tout accès indésirable à l’interface web
- de protéger aussi les endpoints web/API d’administration
- d’avoir un fonctionnement simple, inspiré d’outils comme OpenClaw

Je veux un système basé sur :
- un **token maître unique**
- auto-généré au **tout premier démarrage**
- affiché **une seule fois dans les logs**
- ensuite **jamais réaffiché**
- saisi via une page web dédiée
- puis stockage de l’authentification en **session serveur**

Je veux aussi préparer le support du **HTTPS natif Spring Boot**.

## Objectif global

Implémenter un **MVP sécurité admin** avec :

1. un token maître unique et persistant côté serveur

4. une session HTTP côté serveur après authentification
5. une page `/login` simple
6. un mécanisme de logout
7. une base prête pour l’activation HTTPS côté Spring Boot

# 1. Token maître auto-généré au premier démarrage

## Attendu

Je veux un composant/service dédié qui gère un token admin maître.

Comportement :
- au démarrage, si aucun token n’existe encore :
    - générer un token fort et imprévisible
    - le sauvegarder de manière persistante côté serveur
    - le logguer **une seule fois**
- aux démarrages suivants :
    - recharger le token existant
    - ne jamais le réafficher dans les logs

## Contraintes

- le token doit survivre aux redémarrages
- il doit être stocké dans un emplacement local persistant
- ne pas le mettre en dur dans `application.properties`
- ne pas le committer
- ne pas le réafficher si déjà existant

## Proposition attendue

Créer quelque chose du style :
- `AdminTokenService`
- `AdminTokenInitializer`

Ou une structure équivalente propre.

## Génération du token

Je veux un token fort :
- généré avec un mécanisme sûr (`SecureRandom`)
- suffisamment long
- sous forme texte exploitable à la saisie

Exemple acceptable :
- base64 url-safe
- ou hex

## Emplacement de stockage

Je te laisse choisir l’emplacement le plus cohérent dans le projet, mais il doit être :
- persistant
- simple
- hors git
- facilement compréhensible

Par exemple un fichier dans un répertoire data/runtime/security ou équivalent.

Je veux que tu choisisses une solution simple et saine.

# 2. Authentification web via page de saisie

## Attendu

Créer une page web de login par token.

Routes souhaitées :
- `GET /login`
- `POST /login`
- `POST /logout`

## Comportement

### GET /login
- affiche un formulaire simple
- champ unique : token
- message d’erreur si précédent échec
- si déjà authentifié, peut rediriger vers `/admin`

### POST /login
- récupère le token saisi
- compare au token maître
- si OK :
    - crée une session HTTP authentifiée
    - redirige vers `/admin`
- si KO :
    - retourne vers `/login`
    - affiche un message d’erreur

### POST /logout
- invalide la session
- redirige vers `/login`

## Important

Je veux un système basé sur **session serveur**, pas sur stockage du token dans le navigateur.

Donc :
- en session, on stocke un état du type `adminAuthenticated=true`
- éventuellement quelques métadonnées utiles, mais rien d’excessif

# 3. Filtre global de protection

## Attendu

Créer un filtre Spring (par exemple `OncePerRequestFilter`) qui protège l’application.

## Règle générale

Toute requête vers les zones protégées doit nécessiter une session authentifiée.

## Zones à protéger

Je veux protéger au minimum :
- `/admin/**`
- `/api/admin/**`

Et de manière cohérente, tu peux aussi protéger d’autres routes web/API sensibles si elles doivent l’être.

## Routes publiques à laisser accessibles

Au minimum :
- `/login`
- `/error`
- ressources statiques :
    - `/css/**`
    - `/js/**`
    - `/img/**`
    - `/webjars/**` si pertinent

## Comportement du filtre

Si requête vers zone protégée et utilisateur non authentifié :
- si c’est une page web classique :
    - redirection vers `/login`
- si c’est un endpoint API :
    - retour `401 Unauthorized`

Je veux une distinction propre entre navigation web et appel API.

## Détection session

Le filtre doit s’appuyer sur la session HTTP et l’état d’authentification mis après login.

# 6. Préparer le HTTPS natif Spring Boot

## Important

Je veux préparer le terrain pour activer HTTPS directement dans Spring Boot, sans Nginx pour l’instant.

Je ne te demande pas forcément de fournir un vrai certificat prêt à l’emploi, mais je veux :

1. une structure de configuration propre
2. une documentation minimale dans le code ou un commentaire de synthèse final
