Fonctionnement global

Premier démarrage:
AdminTokenInitializer initialise le token maître.
Si token absent/invalide:
AdminTokenService génère un token fort (32 bytes SecureRandom, Base64 URL-safe), le persiste, et le loggue une seule fois.
Démarrages suivants:
token relu depuis fichier, jamais reloggé.
Login:
POST /login compare le token sans le logguer, puis crée une session serveur (adminAuthenticated=true).
Logout:
POST /logout invalide la session.
Protection:
AdminAuthFilter protège les routes admin via session.
Web non auth -> redirection /login
API non auth -> 401 Unauthorized JSON

Après relecture :
Faiblesses / points discutables identifiés

Ordre des filtres implicite
Sans @Order, le couplage entre filtres pouvait devenir fragile lors de futures phases.

Clés de session trop génériques
adminAuthenticated / adminAuthenticatedAt risquaient des collisions futures dans une appli qui grossit.

Niveau de log du premier token
Le log en WARN pour un événement normal (premier bootstrap) créait un faux signal d’alerte ops.

Dette mineure restante (non bloquante)
Le package historique controler (typo) reste incohérent avec controller, mais c’est transverse et hors périmètre MVP actuel.

Tests bons mais encore unitaires
Ils couvrent bien la logique métier du MVP, mais pas encore un scénario intégré complet HTTP + chaîne de filtres.

Corrections utiles proposées

Rendre l’ordre des filtres explicite.
Namespacer les clés de session admin.
Passer le log “token généré une seule fois” en INFO.
Ne pas lancer maintenant de refactor global de packages ni de refonte de tests intégration (hors périmètre).

Résumé final propre

La séparation configuration/runtime est bonne pour ce MVP: config via properties, état runtime via session + fichier token persistant.
Le modèle est simple et cohérent: TokenService (secret), AuthenticationService (session), AuthFilter (garde d’accès), LoginController (flux web).
Le découplage vis-à-vis orchestrator/mémoire/tooling/policy/workspace est respecté: la sécurité admin est isolée dans son module et ne tire pas ces couches.
La dette restante est maîtrisée: typo historique de package, absence de test intégré bout-en-bout (non bloquant pour cette phase).
Aucune fonctionnalité hors périmètre n’a été ajoutée.
