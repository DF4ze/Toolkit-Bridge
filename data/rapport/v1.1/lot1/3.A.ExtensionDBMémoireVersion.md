# 🔹 Phase 3A — Versioning JPA + préparation de la configuration mémoire en DB

La Phase 1 et la Phase 2 ont été implémentées.
Le runtime métier est désormais DB-only.
Le bootstrap est explicite, non destructif, et limité au démarrage.

Je veux maintenant réaliser la **Phase 3A**, sans encore brancher complètement le runtime mémoire sur la DB.

## 🎯 Objectif

- introduire une protection d’écriture par `@Version` sur la configuration administrable stockée en DB
- préparer proprement la future lecture DB de `memory.configuration`
- garder le runtime actuel stable
- ne pas encore migrer les services mémoire métier vers cette configuration DB

## Décisions d’architecture déjà actées

- DB = source de vérité pour la configuration administrable
- runtime métier = DB-only
- bootstrap = démarrage uniquement
- pas de fallback runtime YAML
- pas de reload implicite
- mémoire conversationnelle = reste en RAM
- seule la configuration mémoire est concernée par cette phase, pas les données mémoire ni le store conversationnel

## Travail demandé

### 1. Introduire `@Version` sur l’entité de configuration administrable

Dans :
- `AdministrableConfigurationEntity`

Je veux :
- un champ `version` adapté à JPA
- l’annotation `@Version`
- la persistance correcte de cette information
- une implémentation propre, minimale, sans complexifier inutilement le modèle

Objectif :
- protéger les écritures concurrentes sur les configurations administrables

### 2. Vérifier les impacts minimaux de `@Version`

Analyse et ajuste si nécessaire :
- repository
- store service
- gateway
- sérialisation éventuelle
- tests existants

⚠️ Important :
- ne mets pas encore en place une grosse gestion UX/API des conflits
- si une gestion minimale d’exception est nécessaire pour éviter un comportement brutal, elle peut être ajoutée, mais reste dans un périmètre très sobre

### 3. Préparer la lecture DB de `memory.configuration`

Je veux une préparation propre pour la future Phase 3B.

Cela signifie :
- identifier où la configuration mémoire devra être lue depuis la DB
- introduire si nécessaire un point d’accès clair côté configuration administrable pour `memory.configuration`
- préparer les types / DTO / payloads / services utilitaires nécessaires
- mais sans encore brancher les services mémoire métier sur cette config

⚠️ Important :
- ne remplace pas encore les `@ConfigurationProperties` mémoire dans les services métier
- ne change pas encore le comportement runtime mémoire
- prépare seulement le terrain proprement

### 4. Vérifier la cohérence des payloads mémoire existants

Si `MemoryConfigurationPayload` ou les structures associées doivent être légèrement ajustées pour être une vraie base stable de future lecture DB :
- fais-le
- mais sans élargir le scope
- pas de refonte inutile

### 5. Tests

Je veux des tests utiles sur :
- le versioning JPA de l’entité administrable
- la non-régression des écritures/l ectures DB de configuration
- la disponibilité propre de `memory.configuration` côté configuration administrable
- sans tests artificiels ni trop couplés à l’implémentation interne
