# 🔹 Phase 3B — Brancher le runtime mémoire sur `memory.configuration` (DB-first)

La Phase 3A et la correction 3A.1 ont été réalisées.
Le projet dispose maintenant :
- d’un accès administrable DB à `memory.configuration`
- d’un bootstrap explicite et non destructif
- d’un runtime global DB-only
- d’un `@Version` sur la configuration administrable

Je veux maintenant réaliser la **Phase 3B**.

## 🎯 Objectif

Brancher réellement les services mémoire sur `memory.configuration` stocké en DB, **sans toucher à la mémoire conversationnelle en RAM**, **sans introduire de reload runtime**, et **sans fallback YAML runtime**.

L’objectif est de faire de la **configuration mémoire** une vraie configuration administrable DB-first, tout en gardant les **services métier mémoire** stables, lisibles et correctement découplés.

## Décisions d’architecture déjà actées

- DB = source de vérité pour la configuration administrable
- runtime métier = DB-only
- bootstrap = démarrage uniquement
- pas de fallback runtime YAML
- pas de reload implicite ni hot reload
- mémoire conversationnelle = reste en RAM
- les données mémoire (facts, rules, episodes, etc.) ne sont pas à refondre dans cette phase
- seule la **configuration mémoire** est concernée par cette phase

## Travail demandé

### 1. Identifier précisément les services mémoire à brancher sur la configuration DB

Je veux que tu identifies les services actuellement alimentés par :
- `MemoryIntegrationProperties`
- `MemoryRetrievalProperties`
- `MemoryScoringProperties`
- `ContextAssemblerProperties`
- `GlobalContextProperties`
- éventuellement d’autres propriétés mémoire pertinentes

Tu dois déterminer :
- lesquels doivent désormais consommer `memory.configuration`
- lesquels doivent rester inchangés dans cette phase
- quels points nécessitent une adaptation minimale d’injection ou de résolution

⚠️ Important :
- `ConversationMemoryProperties` et la mémoire conversationnelle RAM ne doivent pas être migrées dans cette phase
- ne touche pas au `InMemoryConversationMemoryStore`

### 2. Introduire une résolution claire de la configuration mémoire depuis la DB

Je veux une solution propre et stable pour le runtime.

Tu peux par exemple introduire un composant dédié du type :
- resolver
- provider
- runtime snapshot loader
- ou autre nom propre si plus pertinent

Mais l’idée est la suivante :
- le runtime mémoire ne doit pas aller lire le YAML
- il doit récupérer sa configuration depuis `memory.configuration`
- cette lecture doit être centralisée et testable
- elle doit éviter de disperser des lectures DB partout dans les services mémoire

### 3. Définir le comportement en cas d’absence de `memory.configuration`

Comme le bootstrap existe déjà, l’absence devrait être rare.
Mais je veux un comportement explicite, propre et robuste.

Tu dois proposer ce qui est le plus sain dans l’architecture actuelle :
- soit exiger que la config soit présente en DB
- soit tolérer une configuration vide en construisant un payload par défaut cohérent
- soit une autre option propre si elle est mieux justifiée

⚠️ Important :
- pas de fallback YAML runtime
- pas de comportement magique implicite mal documenté

### 4. Brancher les services mémoire métier sur cette configuration résolue

Adapter les services concernés pour qu’ils utilisent la configuration mémoire résolue depuis la DB, sans refonte inutile.

Objectif :
- remplacer la dépendance runtime aux `@ConfigurationProperties` mémoire
- garder une architecture propre, lisible et testable
- limiter les changements transverses

### 5. Conserver la séparation avec la mémoire conversationnelle

Je rappelle explicitement :
- la mémoire conversationnelle reste en RAM
- son store ne doit pas être modifié
- son cycle de vie actuel ne doit pas être refondu ici

Si certains paramètres conversationnels existent dans `memory.configuration` mais ne doivent pas être branchés maintenant, signale-le clairement et laisse-les de côté dans cette phase.

### 6. Vérifier la stabilité du modèle `MemoryConfigurationPayload`

Si le payload doit être légèrement ajusté pour correspondre proprement aux besoins runtime réels des services mémoire :
- fais-le
- mais sans refonte excessive
- sans casser inutilement le bootstrap Phase 2
- et sans introduire une structure floue

### 7. Tests

Je veux des tests réellement utiles sur :
- la résolution DB de `memory.configuration`
- le comportement si la config est absente ou incomplète
- les services mémoire branchés sur la config DB
- la non-régression de la mémoire conversationnelle RAM
- l’absence de fallback YAML runtime

Pas de tests artificiels ni trop couplés à l’implémentation interne.

## Contraintes

- N’implémente pas de reload runtime
- N’introduis pas de fallback YAML runtime
- Ne modifie pas la mémoire conversationnelle RAM
- Ne refonds pas les données mémoire persistées
- N’ajoute pas de nouvelle UI
- N’introduis pas de logique ad hoc dispersée
- Garde une architecture Spring Boot propre, lisible, progressive et testable
