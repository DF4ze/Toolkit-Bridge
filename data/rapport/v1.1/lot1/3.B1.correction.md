# 🔹 Phase 3B.1 — Corrections ciblées après relecture d’architecture mémoire

La Phase 3B a été implémentée et relue.
Je veux maintenant un petit lot correctif, strictement limité aux corrections d’architecture réellement pertinentes, sans ajouter de nouvelle fonctionnalité.

## Objectif

Consolider proprement la migration DB-first de la configuration mémoire avant de poursuivre les phases suivantes.

## Corrections à appliquer

### 1. Centraliser les defaults mémoire dans une source unique

Aujourd’hui, les defaults mémoire semblent exister à la fois :
- dans le seed/bootstrap
- et dans le runtime

Je veux :
- une seule source claire des defaults mémoire
- réutilisée par le seed et par le runtime
- pour éviter toute dérive silencieuse

⚠️ Important :
- pas de fallback YAML runtime
- il s’agit d’une source interne de defaults, pas d’un retour au YAML en exécution

### 2. Casser le couplage direct entre runtime mémoire et payload admin

Le runtime mémoire ne doit pas dépendre directement du DTO/payload d’administration.

Je veux :
- un petit mapper / adaptateur / traducteur dédié
- entre `MemoryConfigurationPayload` et le modèle runtime mémoire
- avec une responsabilité claire et testable

Objectif :
- préserver l’autonomie du runtime mémoire
- éviter qu’une évolution admin casse directement le runtime

### 3. Déplacer la validation hors des hot paths

Si certaines validations de configuration sont actuellement faites à chaque exécution métier :
- déplace-les au moment de la construction / initialisation du snapshot runtime
- pour que la config soit validée une fois
- et non à chaque usage

### 4. Clarifier le rôle des `*Properties` mémoire restantes

Si certaines `@ConfigurationProperties` mémoire restent activées dans des modules runtime alors qu’elles ne servent plus réellement qu’au seed/bootstrap :
- clarifie cette séparation
- retire l’ambiguïté côté runtime si c’est possible proprement
- garde une structure Spring Boot lisible

⚠️ Important :
- ne casse pas le seed existant
- ne réintroduis pas de fallback runtime YAML

### 5. Ajouter un test d’intégration minimal sur l’ordre de démarrage

Je veux un test utile qui verrouille explicitement :
- bootstrap de configuration administrable
- puis initialisation du snapshot mémoire runtime

Objectif :
- prouver que l’ordre de démarrage attendu est bien respecté
- éviter une régression silencieuse plus tard

## Contraintes

- ne pas modifier le périmètre fonctionnel de la Phase 3B
- ne pas ajouter de reload
- ne pas toucher à la mémoire conversationnelle RAM
- ne pas refondre les données mémoire persistées
- ne pas ajouter d’UI
- ne pas introduire de logique ad hoc
- rester sobre et ciblé
