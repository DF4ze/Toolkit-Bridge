# 🔹 Micro-lot de stabilisation final — ordre de démarrage réel + contrat mémoire explicite

Contexte

Le socle DB-first a été largement stabilisé :
- runtime DB-only
- bootstrap explicite non destructif
- resolver mémoire centralisé
- mémoire conversationnelle RAM conservée
- séparation bootstrap / runtime / DTO admin clarifiée

Un mini-audit a identifié quelques points résiduels.
Je veux maintenant corriger **uniquement les 2 points prioritaires**, sans rouvrir un chantier plus large.

## 🎯 Objectif

Finaliser la stabilisation du socle en :
1. renforçant la preuve de la séquence réelle `bootstrap -> DB -> resolver`
2. clarifiant explicitement le contrat entre `memory.configuration` et le runtime mémoire

## Travail demandé

### 1. Renforcer le test d’intégration sur l’ordre de démarrage avec une chaîne plus réelle

Le test actuel valide l’ordre de démarrage, mais reste partiellement mocké.

Je veux un test plus proche de la chaîne réelle :
- seed/bootstrap écrit ou complète la DB
- puis le resolver runtime mémoire lit effectivement la configuration depuis la DB
- sans mock inutile sur la lecture finale DB

Objectif :
- prouver explicitement la séquence réelle `bootstrap -> DB -> resolver`
- éviter qu’un futur changement laisse croire que l’ordre est correct alors que le resolver ne lit pas réellement la DB attendue

⚠️ Important :
- rester sobre
- pas besoin d’un test énorme ou lent inutilement
- pas de refonte du bootstrap
- pas de nouvelle fonctionnalité

### 2. Clarifier explicitement les champs `memory.configuration` non encore appliqués au runtime

Aujourd’hui, certains champs du payload admin mémoire existent mais ne sont pas encore consommés côté runtime, notamment dans `integration`.

Je veux :
- documenter explicitement quels champs existent dans `memory.configuration` mais ne sont pas encore appliqués au runtime
- rendre cela visible dans le code au bon endroit :
    - soit dans le mapper
    - soit dans le modèle runtime
    - soit dans le payload / JavaDoc
    - soit via une combinaison sobre de ces éléments
- ajouter un test qui verrouille ce comportement pour éviter une ambiguïté future

Objectif :
- éviter qu’un champ “éditable” soit interprété comme “forcément actif en runtime”
- clarifier le contrat actuel sans introduire de logique ad hoc

⚠️ Important :
- ne pas brancher maintenant ces champs au runtime
- ne pas élargir le scope
- ne pas introduire de comportement implicite

## Contraintes

- ne corriger que ces 2 points
- ne pas lancer un nouveau chantier d’architecture
- ne pas déplacer encore les `*Properties` mémoire dans d’autres packages
- ne pas enrichir maintenant toute la stratégie d’erreurs
- ne pas modifier la mémoire conversationnelle RAM
- ne pas ajouter de reload
- ne pas réintroduire de fallback YAML runtime
