# 🔹 Reload A — Pré-analyse du modèle de reload explicite

Contexte

Une grosse stabilisation du socle a déjà été réalisée :

- configuration administrable DB-first
- suppression du fallback runtime YAML
- bootstrap explicite et non destructif
- `@Version` sur la configuration administrable
- configuration mémoire branchée sur un resolver runtime centralisé
- séparation claire entre :
    - bootstrap / seed
    - runtime
    - DTO admin
    - mémoire conversationnelle RAM

Je veux maintenant préparer un **système de reload explicite**, qui sera plus tard branché à l’UI, mais **pas dans cette phase**.

---

## 🎯 Objectif de cette phase

Je veux uniquement une **pré-analyse d’architecture**, sans implémentation.

Le but est de définir proprement :
- quels domaines sont réellement reloadables maintenant
- comment structurer un mécanisme de reload explicite
- comment garantir un reload sûr, lisible et testable
- sans introduire de reload automatique ni de couplage UI

---

## Décisions d’architecture déjà actées

- Le reload doit être **explicite uniquement**
- Aucun reload automatique après écriture DB
- La stratégie de reload doit être **rebuild**, pas patch live
- En cas d’échec du reload :
    - on conserve l’ancienne version active
    - on ne laisse jamais un domaine dans un état partiellement rechargé
- Le système doit exposer un **résultat structuré**
- L’UI ne sera branchée que plus tard
- Cette phase ne doit pas implémenter le reload
- Cette phase ne doit pas créer d’endpoints UI/web

---

## Périmètre attendu de l’analyse

Je veux que tu analyses le dépôt actuel pour préparer un reload explicite au moins sur les domaines suivants :

1. **Memory runtime configuration**
2. **LLM providers / registry**

Tu peux aussi signaler d’autres domaines potentiellement reloadables plus tard si c’est pertinent, mais sans élargir cette phase.

---

## Questions auxquelles je veux une réponse claire

### 1. Quels composants/classes sont réellement concernés par un futur reload memory ?

Je veux que tu identifies :
- le resolver/snapshot mémoire actuel
- les services qui dépendent de lui
- où se situe aujourd’hui la “référence active”
- ce qu’il faudrait reconstruire et remplacer lors d’un reload

### 2. Quels composants/classes sont réellement concernés par un futur reload LLM ?

Je veux que tu identifies :
- où le registry/provider LLM est construit
- où il est injecté et consommé
- si le modèle actuel permet un remplacement propre du registry actif
- quels points sont sensibles

### 3. Quel contrat de reload recommandes-tu ?

Je veux une proposition propre et sobre sur :
- l’interface de service interne de reload
- le format de résultat
- la façon de représenter un domaine reloadable
- la stratégie de journalisation
- la stratégie d’échec

⚠️ Important :
- pas de sur-ingénierie
- pas de moteur générique inutilement complexe
- une architecture simple, extensible, propre

### 4. Quelle stratégie de remplacement runtime recommandes-tu ?

Je veux une analyse explicite sur :
- rebuild complet du domaine
- remplacement atomique de la référence active
- gestion de l’ancienne version active en cas d’échec
- points de synchronisation/concurrence éventuels

### 5. Quels risques ou dettes vois-tu si on implémente ce système maintenant ?

Je veux que tu identifies :
- risques de couplage
- risques de race condition
- risques de design trop abstrait
- risques de casser le fonctionnement actuel
- zones du code qui demanderaient une petite adaptation préparatoire avant un vrai reload
