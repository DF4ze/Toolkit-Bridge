# 🔹 Reload A.1 — Fondation du mécanisme de reload explicite

Contexte

Une pré-analyse d’architecture a été réalisée pour préparer un système de reload explicite.
Le cadrage suivant est déjà validé :

- reload explicite uniquement
- aucun reload automatique après écriture DB
- stratégie de reload = **rebuild**, pas patch live
- en cas d’échec :
    - on conserve l’ancienne version active
    - aucun état partiellement rechargé ne doit être exposé
- résultat structuré attendu
- pas de couplage UI dans cette phase
- le premier domaine concret sera **memory** dans la phase suivante
- **LLM viendra après**, dans un lot séparé

Je veux maintenant implémenter un **mini-lot de fondation** avant les reloads concrets.

---

## 🎯 Objectif

Poser la **colonne vertébrale interne** du mécanisme de reload explicite, de façon simple, propre et extensible, **sans encore recharger réellement un domaine**.

Cette phase doit fournir :
- les types communs
- le contrat interne
- l’orchestrateur léger
- la structure permettant de brancher ensuite `memory`, puis `llm`

Mais elle ne doit pas encore :
- implémenter un reloader concret memory
- implémenter un reloader concret llm
- modifier le runtime actuel
- ajouter d’endpoint
- ajouter d’UI

---

## Travail demandé

### 1. Définir les types communs du mécanisme de reload

Je veux un modèle simple et stable, sans sur-ingénierie.

À minima, je veux que tu proposes et implémentes des types du style :

- `ReloadDomain`
- `ReloadStatus`
- `ReloadDomainResult`
- `ReloadReport`

Tu peux ajuster les noms si tu trouves mieux, mais reste sobre et cohérent.

### 2. Définir le contrat interne d’un reloader de domaine

Je veux une interface claire du type :
- un reloader sait gérer un domaine précis
- il sait exécuter un reload explicite
- il retourne un résultat structuré

Objectif :
- permettre ensuite de brancher proprement `memory`
- puis `llm`
- sans framework générique complexe

### 3. Implémenter un orchestrateur léger

Je veux un service central du type :
- `ExplicitReloadService`

Responsabilités :
- recevoir une demande de reload sur un ou plusieurs domaines
- trouver les bons reloaders enregistrés
- exécuter les reloads de façon lisible
- produire un `ReloadReport`

⚠️ Important :
- rester simple
- pas de moteur plugin complexe
- une map `domain -> reloader` est suffisante si elle est propre

### 4. Définir la stratégie d’échec dans le contrat

Même s’il n’y a pas encore de reloader concret, je veux que le modèle porte déjà clairement l’intention :

- si un domaine échoue, son résultat doit l’indiquer explicitement
- le rapport global doit être compréhensible
- rien dans cette phase ne doit supposer un patch partiel ou un état intermédiaire

### 5. Journalisation minimale utile

Je veux une base de logs sobre pour les futurs reloads :
- début
- succès
- échec

Mais sans bruit inutile.

### 6. Préparer les phases suivantes sans les implémenter

Cette phase doit préparer proprement :
- Reload B = memory
- Reload C = llm

Mais sans coder les reloaders réels.

Tu peux si besoin :
- ajouter un reloader “stub” de test
- ou un faux reloader de démonstration
- uniquement si cela aide vraiment à tester l’orchestrateur

---

## Questions d’architecture à traiter avant code

Avant d’implémenter, analyse dans le dépôt actuel :

1. où placer le mécanisme de reload pour qu’il reste neutre et réutilisable
2. quelle structure simple tu proposes pour le rapport de reload
3. comment gérer proprement :
    - domaine demandé sans reloader enregistré
    - domaine supporté mais échec pendant reload
4. s’il faut autoriser plusieurs domaines dans un seul appel dès maintenant
5. quel plan d’implémentation tu proposes en 5 à 10 étapes maximum

## Contraintes

- N’implémente aucun reloader memory

- Ne modifie pas le runtime actuel
- N’ajoute aucun endpoint web/admin
- N’ajoute aucune UI
- Ne crée pas de framework générique compliqué
- Reste simple, testable, lisible
