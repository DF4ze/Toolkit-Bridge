# 🔹 Reload B — Implémentation du reload explicite du runtime mémoire

Contexte

Le socle de reload explicite a été préparé et stabilisé :

- types communs de reload en place
- orchestrateur `ExplicitReloadService` en place
- contrat `ReloadDomainResult` durci
- tests d’invariants et de comportement de l’orchestrateur en place

Le runtime mémoire est déjà DB-first :
- configuration mémoire stockée en DB via `memory.configuration`
- resolver central runtime mémoire déjà en place
- snapshot mémoire distinct du bootstrap
- pas de fallback YAML runtime
- mémoire conversationnelle RAM explicitement hors périmètre

Je veux maintenant implémenter **Reload B**, c’est-à-dire le **premier reloader concret**, sur le domaine mémoire uniquement.

---

## 🎯 Objectif

Permettre un reload explicite du **runtime mémoire**, basé sur un **rebuild complet**, puis un **swap atomique** de la configuration active, avec conservation de l’ancienne version en cas d’échec.

## Décisions d’architecture déjà actées

- reload explicite uniquement
- aucun reload automatique après écriture DB
- stratégie = **build / validate / swap**
- en cas d’échec :
    - on conserve l’ancienne version active
    - aucun état partiellement rechargé ne doit être visible
- pas de fallback YAML runtime
- pas de patch live sur objets partagés
- mémoire conversationnelle RAM reste inchangée
- pas d’endpoint / pas d’UI dans cette phase

---

## Travail demandé

### 1. Introduire un reloader concret pour le domaine mémoire

Je veux un handler/reloader dédié au domaine mémoire, branché sur le socle reload existant.

Exigences :
- il doit reconstruire complètement la configuration runtime mémoire à partir de la DB
- il doit valider la nouvelle configuration avant tout swap
- il ne doit exposer la nouvelle version qu’après validation complète
- il doit retourner un résultat structuré compatible avec le mécanisme de reload existant

### 2. Réutiliser proprement l’architecture mémoire existante

Tu dois t’appuyer sur :
- `MemoryRuntimeConfigurationResolver`
- `MemoryRuntimeConfiguration`
- `MemoryRuntimeConfigurationMapper`
- `AdministrableConfigurationGateway.loadMemoryConfiguration()`

Mais sans bricoler une deuxième logique parallèle.

Objectif :
- factoriser proprement ce qui doit l’être
- éviter de dupliquer la logique build/validation
- conserver un design lisible

### 3. Gérer explicitement le cache global context

Point important :
- le cache de `MarkdownSharedGlobalContextProvider` doit être invalidé explicitement lors d’un reload mémoire réussi

Je veux une solution propre :
- pas de hack
- pas de dépendance cachée
- pas d’invalidation opportuniste floue

Si une petite adaptation ciblée est nécessaire sur ce provider pour exposer une invalidation claire, fais-la.

### 4. Garantir la conservation de l’ancienne version active en cas d’échec

Je veux que le reloader :
- construise la nouvelle config dans des variables locales
- valide complètement
- ne fasse le swap qu’à la fin

Si le build ou la validation échoue :
- aucune mutation visible ne doit avoir eu lieu
- l’ancienne config doit continuer à être utilisée

### 5. Clarifier le comportement si la DB est absente ou partielle

Le comportement actuel doit rester cohérent avec la stratégie déjà en place :
- DB-first
- mais robuste si la config manque ou est partielle
- sans fallback YAML runtime

Ne change pas inutilement cette stratégie, sauf si une adaptation ciblée est indispensable pour rendre le reload sûr et cohérent.

### 6. Brancher le reloader sur l’orchestrateur de reload

Le reloader mémoire doit être utilisable via le mécanisme de reload explicite existant, mais sans endpoint/UI.

### 7. Tests

Je veux des tests réellement utiles sur :

- succès d’un reload mémoire
- échec d’un reload mémoire avec conservation de l’ancienne version active
- invalidation du cache global context après reload réussi
- non invalidation si le reload échoue
- intégration correcte avec l’orchestrateur de reload
- absence d’état partiellement rechargé

Pas de tests artificiels. Pas de surcouplage à l’implémentation interne.

# 🔹 Reload B.1 — Corrections ciblées de robustesse et de contrat

Reload B a été implémenté et relu.
Je veux maintenant un micro-lot correctif, strictement limité aux points de robustesse et de maintenabilité réellement pertinents.

## Objectif

Finaliser proprement le reload mémoire avant de passer au domaine suivant.

## Corrections à appliquer

### 1. Encadrer explicitement le cas d’échec de `invalidateCache()` après reload réussi

Point critique :
- aujourd’hui, le swap runtime peut déjà être fait avant `invalidateCache()`
- si l’invalidation échoue, il peut rester une incohérence entre runtime rechargé et cache global context

Je veux que tu clarifies et implémentes explicitement une stratégie propre sur ce point.

Attendu :
- pas d’état ambigu “reload appliqué mais report incohérent”
- comportement explicite, documenté et testable
- logs clairs

Tu peux choisir l’approche la plus saine, mais elle doit être cohérente et assumée dans le code.

### 2. Réduire le couplage du handler au type interne du resolver

Je veux éviter que le handler dépende directement d’un record imbriqué interne du resolver si une alternative simple et propre existe.

Attendu :
- soit un petit type plus propre/externe
- soit un contrat plus simple
- sans sur-ingénierie

### 3. Rendre les tests moins fragiles aux messages exacts

Dans les tests du handler :
- vérifier surtout les statuts, domaines et effets utiles
- éviter une dépendance trop forte au texte exact des messages quand ce n’est pas indispensable

### 4. Clarifier la sémantique de `invalidateCache()`

Je veux que la Javadoc ou le contrat précise explicitement :
- si l’appel est attendu comme best effort
- s’il doit éviter de throw
- ou au contraire si une exception est autorisée et comment elle est traitée

Le contrat doit être clair pour les futurs reloaders/mainteneurs.

### 5. Optionnel si très simple : améliorer un peu la lisibilité des noms

Si le renommage du champ `snapshot` vers quelque chose de plus explicite est trivial et améliore la lecture, tu peux le faire.
Sinon, ne force pas ce point.

## Contraintes

- ne pas commencer Reload C
- ne pas modifier la mémoire conversationnelle
- ne pas ajouter d’endpoint
- ne pas ajouter d’UI
- ne pas faire de refonte large
- rester dans un micro-lot ciblé
