# 🔹 Reload C — Pré-analyse du reload explicite du runtime LLM

Contexte

Le socle de reload explicite est en place.
Le domaine mémoire a déjà été rendu reloadable avec :
- rebuild complet
- validation
- swap atomique
- conservation de l’ancienne version active en cas d’échec

Je veux maintenant préparer le **reload explicite du runtime LLM**, mais dans cette phase je veux **uniquement une pré-analyse**, sans implémentation.

---

## 🎯 Objectif

Définir proprement comment rendre le domaine **LLM** reloadable, sans casser l’architecture existante, et sans introduire de hack lié au cycle de vie Spring.

Le futur reload devra permettre :
- de reconstruire complètement le registry/providers LLM à partir de la DB
- de valider cette reconstruction
- de remplacer atomiquement la référence active
- de conserver l’ancienne version active en cas d’échec
- sans redémarrage
- sans reload automatique
- sans couplage UI

---

## Décisions d’architecture déjà actées

- reload explicite uniquement
- aucun reload automatique après écriture DB
- stratégie = **build / validate / swap**
- en cas d’échec :
    - l’ancienne version active reste utilisée
    - aucun état partiellement rechargé ne doit être exposé
- pas de fallback YAML runtime
- pas d’endpoint / pas d’UI dans cette phase
- le socle reload (`ExplicitReloadService`, `ReloadDomain`, etc.) existe déjà
- Reload C ne doit pas modifier la mémoire ni la conversation RAM

## Questions auxquelles je veux une réponse claire

### 1. Où se situe aujourd’hui la référence active LLM ?

Je veux que tu identifies précisément :
- où le `LlmProviderRegistry` est construit
- où il est injecté
- où il est consommé

### 2. Quelle adaptation minimale recommandes-tu pour rendre le runtime LLM reloadable ?

Je veux une proposition propre sur :
- holder runtime dédié (`AtomicReference`, `volatile`, autre si mieux justifié)
- factory/builder du registry
- découplage avec la config Spring actuelle
- stratégie de remplacement atomique

⚠️ Important :
- pas de sur-ingénierie
- pas de framework exotique
- pas de réinjection Spring dynamique compliquée

### 3. Quelle logique de build/validation recommandes-tu ?

Je veux que tu identifies ce qui doit être validé avant swap, par exemple :
- noms de providers
- doublons
- configuration vide
- champs obligatoires
- compatibilité avec l’usage actuel de `DefaultLlmService`

### 4. Quels composants devront probablement évoluer ?

Je veux que tu identifies les classes qui devront être adaptées ou introduites, par exemple :
- holder runtime LLM
- builder/factory du registry
- reloader concret LLM
- adaptation de `DefaultLlmService`
- adaptation éventuelle de `OpenAiLikeConfiguration`

### 5. Quels risques ou pièges vois-tu ?

Je veux une analyse claire sur :
- pièges liés au lifecycle Spring
- risques de fuite de secrets dans les logs
- risques de garder un modèle hybride boot-only / runtime-active
- risques de couplage trop fort entre config Spring et runtime LLM
- risques de tests trop artificiels
