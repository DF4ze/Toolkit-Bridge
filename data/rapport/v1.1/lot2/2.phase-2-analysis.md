# Phase 2 — Analyse d’extraction des queries à faible couplage

- Blocs à extraire (actuellement dans la façade)

  - traces : `listRecentTraces(...)`, `toTraceItem(...)`
  - artefacts : `listRecentArtifacts(...)`, `toArtifactItem(...)`
- Dépendances directes des 3 blocs
  - tâches : `AdminTaskStore` (`src/main/java/fr/ses10doigts/toolkitbridge/service/admin/task/AdminTaskStore.java`)
  - traces : `AgentTraceQueryService` (`src/main/java/fr/ses10doigts/toolkitbridge/service/agent/trace/AgentTraceQueryService.java`)
  - artefacts : `ArtifactService` (`src/main/java/fr/ses10doigts/toolkitbridge/service/agent/artifact/service/ArtifactService.java`)

- DTO manipulés
  - `TechnicalAdminView.TaskItem`
  - `TechnicalAdminView.TraceItem`
  - `TechnicalAdminView.ArtifactItem`
  - source DTO container : `src/main/java/fr/ses10doigts/toolkitbridge/model/dto/admin/technical/TechnicalAdminView.java`
- Usages applicatifs à préserver inchangés
  - `TechnicalAdminController` (REST)
  - `TechnicalAdminPageController` (UI pages)
  - `AdminDashboardService` via `getOverview()` (indirectement concerné car `getOverview()` appelle les 3 méthodes)
- Tests existants impactés

  - `src/test/java/fr/ses10doigts/toolkitbridge/controler/web/admin/technical/TechnicalAdminControllerTest.java`
  - `src/test/java/fr/ses10doigts/toolkitbridge/controler/web/admin/technical/TechnicalAdminPageControllerTest.java`
- Points communs transverses entre les 3 blocs
  - usage de `technicalProperties.sanitizeLimit(...)`
  - patterns tri descendant + `limit(...)` pour traces/artefacts
  - mapping vers DTO `TechnicalAdminView.*`

## 2. Analyse des trois blocs à extraire
- Bloc tâches (`listRecentTasks`)
  - Couplage réel : faible.
  - Dépendances : `AdminTaskStore` + `AdminTechnicalProperties`.
  - Logique : sanitation de limite, query `recent(...)`, mapping `AdminTaskSnapshot -> TaskItem`.
  - Particularités : peu d’orchestration, pas de règles de branchement multiples.
  - Niveau de risque extraction : faible.

- Bloc traces (`listRecentTraces`)
  - Couplage réel : faible à modéré.
  - Dépendances : `AgentTraceQueryService` + `AdminTechnicalProperties`.
  - Logique :
    - branchement source (`recentEvents` vs `recentEventsForAgent` selon `agentId` blank/non blank)
    - tri descendant par `occurredAt`
    - limitation
    - mapping `AgentTraceEvent -> TraceItem`.
  - Particularités : dépend du helper `isBlank`, comportement de tri/limit à conserver strictement.
  - Niveau de risque extraction : faible à modéré.

- Bloc artefacts (`listRecentArtifacts`)
  - Couplage réel : modéré (mais toujours extraction faisable en phase 2).
  - Dépendances : `ArtifactService` + `AdminTechnicalProperties`.
  - Logique :
    - règle de priorité de filtres (contrat implicite verrouillé en phase 1): `taskId` > `agentId` > global
    - tri descendant par `createdAt`
    - limitation
    - mapping `Artifact -> ArtifactItem`.
  - Particularités : plus de branches conditionnelles, risque de régression si ordre de priorité modifié.
  - Niveau de risque extraction : modéré.

## 3. Points sensibles et risques d’extraction
- Risque de duplication
  - `sanitizeLimit`, `isBlank`, patterns tri/limit peuvent être dupliqués dans 3 services.
  - Recommandation phase 2 : accepter une duplication légère locale plutôt qu’introduire une abstraction utilitaire prématurée.

- Risque de déplacer trop de logique UI
  - Mapping vers `TechnicalAdminView.*` est actuellement en façade.
  - Décision actée : garder ce mapping côté query services futurs (pas de DTO intermédiaires dans cette phase).

- Risque de casser les tests
  - `TechnicalAdminFacadeTest` a une couverture comportementale importante sur ces méthodes.
  - Si la façade devient délégante, ces tests devront être partiellement redistribués vers les nouveaux services + tests de délégation façade.

- Risque d’abstraction artificielle
  - Créer une super-interface commune des query services n’apporte pas de valeur ici.
  - Recommandation : 3 services simples, indépendants, orientés bloc.

- Comportements à conserver strictement
  - tâches : filtration `agentId/status` via store + mapping identique.
  - traces : sélection source selon `agentId` blank/non blank, tri desc, limit.
  - artefacts : priorité `taskId` sur `agentId`, puis global; tri desc; limit.
  - `TechnicalAdminFacade` doit rester l’entrée utilisée par contrôleurs.

## 4. Plan d’extraction recommandé
- Ordre recommandé
  1. `AdminTaskQueryService`
  2. `TraceQueryService`
  3. `ArtifactQueryService`

- Déplacement exact par service
  - `AdminTaskQueryService`
    - déplacer `listRecentTasks(...)`
    - déplacer `toTaskItem(...)`
    - dépendances injectées : `AdminTaskStore`, `AdminTechnicalProperties`
  - `TraceQueryService`
    - déplacer `listRecentTraces(...)`
    - déplacer `toTraceItem(...)`
    - inclure helper local `isBlank(...)` (ou équivalent privé local)
    - dépendances injectées : `AgentTraceQueryService`, `AdminTechnicalProperties`
  - `ArtifactQueryService`
    - déplacer `listRecentArtifacts(...)`
    - déplacer `toArtifactItem(...)`
    - inclure helper local `isBlank(...)` (ou équivalent privé local)
    - dépendances injectées : `ArtifactService`, `AdminTechnicalProperties`

- Ce qui doit rester temporairement dans la façade
  - `getOverview()` inchangé en structure, continuant d’appeler `listRecentTasks/traces/artifacts` publics de la façade.
  - façade conserve les signatures publiques actuelles pour compatibilité contrôleurs/tests.
  - blocs non périmètre phase 2 restent en place : agents/runtime/policy/tools, configuration, rétention.

- Helpers: mutualiser ou laisser local
  - phase 2 recommandée : laisser local dans chaque service (`isBlank`, mapping spécifique), pour éviter sur-ingénierie.
  - mutualisation éventuelle seulement si duplication devient coûteuse dans phases ultérieures.

## 5. Impact sur les tests
- Tests à conserver
  - `TechnicalAdminControllerTest` et `TechnicalAdminPageControllerTest` (contrats façade inchangés)
  - `AdminDashboardServiceTest` (dépend de `getOverview`, inchangé en API)

- Tests à adapter
  - `TechnicalAdminFacadeTest`:
    - déplacer les assertions métier spécifiques tâches/traces/artefacts vers tests des nouveaux services
    - garder dans la façade des tests de délégation et de stabilité contrat public
    - conserver les tests `getOverview` (car reste dans la façade)

- Tests à ajouter (probables)
  - `AdminTaskQueryServiceTest` (sanitization + mapping)
  - `TraceQueryServiceTest` (branchements source + tri/limit + mapping)
  - `ArtifactQueryServiceTest` (priorité filtres + tri/limit + mapping)

- Risque principal côté tests
  - conserver trop de logique métier testée dans la façade et la retester dans nouveaux services (duplication coûteuse).
  - stratégie recommandée : déplacer la preuve comportementale au niveau service extrait, laisser façade en tests de délégation/assemblage.

## 6. Recommandation finale
- L’extraction phase 2 est pertinente et à faible risque si elle reste strictement mécanique sur 3 blocs query.
- Le meilleur compromis simplicité/sécurité est :
  - extraire d’abord tâches (le plus simple),
  - puis traces,
  - puis artefacts (bloc le plus branché donc à traiter après sécurisation des patterns).
- Maintenir `TechnicalAdminFacade` comme façade délégante publique sans changer contrôleurs ni DTO.
- Éviter toute abstraction commune prématurée dans cette phase; privilégier des services lisibles et autonomes.
