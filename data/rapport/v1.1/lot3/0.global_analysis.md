Contexte fonctionnel du lot :
Nous voulons introduire une persistance durable minimale et ciblée pour les éléments techniques qui ne doivent pas disparaître au redémarrage.
Ordre de priorité attendu :
1. interventions humaines
2. tâches admin
3. événements critiques / traces utiles au diagnostic
4. éventuellement états de supervision si réellement nécessaire

Attention :
- Il ne faut PAS chercher à persister trop tôt tout le flux d’observabilité.
- Il ne faut PAS généraliser prématurément.
- Il faut distinguer ce qui doit absolument survivre à un redémarrage, ce qui est simplement utile, et ce qui peut rester en mémoire pour l’instant.

Travail demandé :

1. Cartographier précisément l’existant
   Analyse le code et identifie :
- les objets techniques actuellement gardés uniquement en mémoire,
- les services / composants / stores / registries concernés,
- les cycles de vie de ces objets,
- les points de création / mise à jour / lecture / suppression éventuelle,
- les impacts d’un redémarrage sur chacun d’eux.

2. Produire un inventaire structuré
   Pour chaque type d’objet concerné, indiquer au minimum :
- nom de l’objet / concept
- classes principales impliquées
- mode de stockage actuel (in-memory, cache local, collection interne, etc.)
- utilité métier / exploitation / diagnostic
- impact si perte au redémarrage
- criticité de persistance : indispensable / utile / différable
- niveau de difficulté estimé pour persistance
- dépendances ou risques architecturaux

3. Prioriser selon la valeur d’exploitation
   Construis une proposition argumentée de priorisation en respectant l’ordre cible suivant :
1. interventions humaines
2. tâches admin
3. événements critiques / traces utiles
4. états de supervision éventuels

Pour chaque bloc, explique :
- pourquoi il devrait ou non être persisté maintenant,
- quel périmètre minimal serait pertinent,
- ce qui doit explicitement rester hors périmètre à ce stade.

4. Proposer un modèle de persistance minimal
   Sans coder, propose pour chaque bloc prioritaire :
- une stratégie simple de persistance,
- le type de modèle de données attendu,
- les champs minimaux à conserver,
- la granularité recommandée,
- les besoins éventuels d’index / recherche / tri,
- les points d’attention sur cycle de vie, expiration, purge future, rétention future.

5. Identifier les décisions à trancher
   Liste explicitement les questions de conception qui nécessitent un arbitrage humain avant implémentation.
   Exemples possibles :
- faut-il persister l’intégralité d’un objet ou seulement un sous-ensemble utile ?
- faut-il historiser les changements d’état ou seulement l’état courant ?
- faut-il distinguer événement métier, trace technique et audit d’exploitation ?
- faut-il recharger certains objets au démarrage ou seulement les rendre consultables en historique ?

6. Préparer la suite pour un découpage en phases
   Propose un découpage du lot 3 en sous-phases unitaires, cohérentes et sûres pour une implémentation progressive par Codex.
   Chaque sous-phase doit :
- avoir un objectif clair,
- avoir un périmètre limité,
- éviter les refactorings diffus,
- être testable,
- minimiser le risque de dérive.

Contraintes d’analyse :
- Appuie-toi sur le code réellement présent, pas sur des suppositions.
- Cite précisément les classes / services / packages utiles.
- Signale clairement les zones ambiguës ou les conventions peu claires.
- Si certains objets paraissent “persistables” mais en réalité dépendent d’un runtime volatile, précise-le.
- Ne propose pas une architecture trop ambitieuse.
- Favorise la robustesse, la simplicité et la lisibilité.
