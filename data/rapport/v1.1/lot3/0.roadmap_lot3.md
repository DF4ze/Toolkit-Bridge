# Mini-roadmap — Lot 3 : Persistance durable minimale

## 1. Objectif du lot

Introduire une persistance durable ciblée pour les objets techniques critiques,
sans sur-architecturer ni généraliser prématurément.

Le lot doit permettre :
- de ne plus perdre les données critiques au redémarrage,
- de conserver une lisibilité opérationnelle minimale,
- de préparer les futures stratégies de rétention et d’audit.

Contraintes :
- pas de refactor global,
- pas de sur-persistance,
- compatibilité maximale avec l’existant,
- implémentation progressive par blocs indépendants.

---

## 2. Découpage global du lot

Le lot est découpé en 4 phases principales + 1 optionnelle :

1. Interventions humaines (priorité critique)
2. Tâches admin (snapshot exploitation)
3. Traces critiques (diagnostic minimal)
4. Rétention / purge minimale
5. (Optionnel) États de supervision persistés

Chaque phase est conçue pour :
- être indépendante,
- être testable,
- limiter les impacts transverses,
- éviter les dérives architecturales.

---

## 3. Phase 1 — Persistance des interventions humaines

### 🎯 Objectif
Garantir qu’aucune demande d’intervention humaine (PENDING ou traitée)
ne soit perdue au redémarrage.

### 📌 Périmètre
- `HumanInterventionRequest`
- `HumanInterventionDecision`
- `HumanInterventionService`

### 🧱 Travail attendu
- Introduire une persistance durable (JPA / SQLite existant)
- Remplacer ou adapter `InMemoryHumanInterventionService`
- Conserver les signatures existantes (pas de refactor API)
- Supporter :
    - open()
    - findById()
    - findPending()
    - recordDecision()

### ⚠️ Points d’attention
- Le service n’est pas encore branché → ne pas sur-étendre
- Attention aux champs `metadata` (potentiellement sensibles)
- Pas besoin d’audit complet des transitions dans ce lot

### ✅ Critère de sortie
- Une demande créée avant restart est retrouvée après restart
- Les décisions sont conservées
- Les tests couvrent le cycle complet

---

## 4. Phase 2 — Persistance des tâches admin

### 🎯 Objectif
Conserver l’historique des tâches visibles dans Marcel après redémarrage.

### 📌 Périmètre
- `AdminTaskSnapshot`
- `AdminTaskStore`
- `AdminTaskQueryService`

### 🧱 Travail attendu
- Implémenter un store durable pour les snapshots
- Conserver :
    - `taskId`
    - `status`
    - `firstSeen`
    - `lastSeen`
    - `errorMessage`
    - métadonnées utiles
- Remplacer `InMemoryAdminTaskStore` ou introduire une impl hybride

### ⚠️ Points d’attention
- Snapshot ≠ historique complet
- Ne pas historiser toutes les transitions
- Garder le comportement existant (API inchangée)

### ✅ Critère de sortie
- Les tâches restent visibles après restart
- Le tri et les filtres existants fonctionnent
- Pas de duplication incohérente

---

## 5. Phase 3 — Persistance des traces critiques

### 🎯 Objectif
Permettre une consultation stable des événements critiques via l’admin.

### 📌 Périmètre
- `AgentTraceEvent`
- `DefaultAgentTraceService`
- `TraceQueryService`

### 🧱 Travail attendu
- Introduire un stockage “critical traces”
- Filtrer uniquement certains types :
    - `ERROR`
    - `TASK_STARTED`
    - `RESPONSE`
    - `DELEGATION`
- Ajouter un sink dédié OU enrichir l’existant
- Adapter la lecture côté `TraceQueryService`

### ⚠️ Points d’attention
- Ne pas persister tous les événements (volume énorme)
- Ne pas supprimer le JSONL existant
- Bien définir le rôle de chaque source :
    - mémoire → court terme
    - JSONL → brut
    - DB → critique exploitable

### ✅ Critère de sortie
- Les erreurs et événements clés sont visibles après restart
- Le bruit est maîtrisé
- Les performances restent correctes

---

## 6. Phase 4 — Rétention et purge minimale

### 🎯 Objectif
Éviter une croissance infinie des données persistées.

### 📌 Périmètre
- nouvelles tables introduites (interventions, tasks, traces)

### 🧱 Travail attendu
- Implémenter une purge simple basée sur TTL
- S’appuyer sur :
    - `PersistenceRetentionPolicyResolver`
- Définir des durées simples (ex: 7j / 30j)

### ⚠️ Points d’attention
- Pas de système d’archivage complexe
- Pas de stratégie multi-tier
- Ne pas coupler à l’UI

### ✅ Critère de sortie
- Les données expirées sont supprimées
- Le système reste simple et lisible

---

## 7. Phase 5 — (Optionnelle) États de supervision

### 🎯 Objectif
Conserver une trace informative du dernier état connu des agents.

### 📌 Périmètre
- `AgentRuntimeState`
- `AgentRuntimeRegistry`

### 🧱 Travail attendu (si activé)
- Stocker un snapshot “last seen”
- Marquer comme obsolète après restart

### ⚠️ Points d’attention
- Ne jamais restaurer un état actif (`busy`)
- Risque de confusion élevé

### ✅ Recommandation
👉 À faire uniquement si besoin explicite  
👉 Sinon à ignorer dans ce lot
