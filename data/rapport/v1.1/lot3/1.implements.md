# Prompt Codex — Implémentation Phase 1 — Persistance des interventions humaines

Tu travailles sur le projet Toolkit-Bridge / Marcel.

## 🎯 Objectif

Implémenter la persistance durable des interventions humaines,
en respectant STRICTEMENT :

- le rapport d’analyse Phase 1
- les décisions d’architecture validées
- un périmètre minimal (pas de sur-implémentation)

---

## ⚠️ Contraintes absolues

- ❌ Pas de refactor global
- ❌ Pas de modification des APIs publiques existantes
- ❌ Pas d’ajout de fonctionnalités hors périmètre
- ❌ Pas d’historisation complète
- ❌ Pas de cache mémoire complexe

- ✅ Implémentation simple, robuste, lisible
- ✅ Compatible SQLite / JPA existant
- ✅ Respect strict du contrat `HumanInterventionService`

---

## 📌 Décisions validées à respecter

1. Persistance = état courant + décision finale uniquement
2. Table unique (pas de table séparée pour décision)
3. `metadata` :
    - stockée en JSON
    - taille limitée
    - léger filtrage (clé type "secret", "token", etc.)
4. Concurrence :
    - update conditionnel `WHERE status = PENDING`
5. Lecture :
    - directe depuis DB (pas de cache)
6. `decision_metadata` :
    - conservée (nullable)

---

## 🧱 Travail à réaliser

### 1. Créer le modèle de persistance

Créer une entité JPA :

`HumanInterventionEntity` (ou nom équivalent cohérent)

Champs obligatoires :

- requestId (unique)
- createdAt
- agentId
- sensitiveAction
- kind
- status
- summary

Champs optionnels :

- traceId
- detail
- requestMetadataJson (TEXT)

Champs décision :

- decisionStatus (nullable)
- decisionDecidedAt (nullable)
- decisionActorId (nullable)
- decisionChannel (nullable)
- decisionComment (nullable)
- decisionMetadataJson (nullable)

Contraintes :
- enums stockées en STRING
- JSON en TEXT
- index :
    - request_id unique
    - (status, created_at)

---

### 2. Créer le repository

Créer :

`HumanInterventionRepository`

Fonctions minimales :

- findByRequestId
- findByStatusOrderByCreatedAtAsc
- save
- updateDecisionIfPending (IMPORTANT)

👉 Cette méthode doit faire :
UPDATE avec condition :
- request_id = ?
- status = 'PENDING'

Retourner :
- nombre de lignes modifiées

---

### 3. Implémenter le service persistant

Créer :

`PersistentHumanInterventionService`
(implémentation de `HumanInterventionService`)

⚠️ Ne PAS modifier l’interface existante

---

#### 3.1 open(...)

- générer UUID
- createdAt = now
- status = PENDING
- mapper Draft → Entity
- gérer metadata :
    - sérialisation JSON
    - limitation taille
    - filtrage simple (clé contenant "secret", "token", etc.)
- save
- notifier `onRequestOpened`

---

#### 3.2 findById(...)

- lire via repository
- mapper Entity → Domain (`HumanInterventionRequest`)

---

#### 3.3 findPending()

- query DB : status = PENDING
- tri par createdAt ASC
- mapper vers domaine

---

#### 3.4 recordDecision(...)

⚠️ POINT CRITIQUE

- appeler `updateDecisionIfPending`
- si 0 ligne → retourner Optional.empty()
- sinon :
    - récupérer l’entité
    - mapper vers domaine
    - notifier `onDecisionRecorded`

👉 Respect strict :
- aucune transition si déjà terminal
- comportement identique à l’actuel

---

### 4. Gestion du JSON metadata

Implémenter :

- sérialisation via ObjectMapper
- limitation taille (ex: 4KB)
- filtrage simple :
    - supprimer clés contenant :
        - "token"
        - "secret"
        - "password"

⚠️ Pas de logique complexe

---

### 5. Mapper Entity ↔ Domain

Créer un mapper propre :

- Entity → `HumanInterventionRequest`
- Draft → Entity
- Decision → update Entity

⚠️ Ne pas polluer le service avec mapping inline

---

### 6. Configuration Spring

- Remplacer `InMemoryHumanInterventionService`

⚠️ Optionnel :
- garder l’ancienne impl pour tests si nécessaire
