# Prompt Codex — Implémentation Phase 2 — Persistance des tâches admin

Tu travailles sur le projet Toolkit-Bridge / Marcel.

## 🎯 Objectif

Implémenter la persistance durable des tâches admin,
en respectant strictement le périmètre de la phase 2 du lot 3 :

> conserver uniquement le dernier snapshot par `taskId`

---

## ⚠️ Contraintes absolues

- ❌ Pas de refactor global
- ❌ Pas de modification des APIs publiques existantes
- ❌ Pas d’historisation complète des transitions
- ❌ Pas de purge/rétention dans cette phase
- ❌ Pas de cache mémoire
- ❌ Pas de changement de comportement côté orchestrateur

- ✅ Implémentation simple, robuste, lisible
- ✅ Compatible SQLite / JPA existant
- ✅ Respect strict du contrat `AdminTaskStore`
- ✅ Conservation du comportement fonctionnel existant

---

## 📌 Décisions validées à respecter

1. On persiste uniquement le **dernier snapshot par `taskId`**
2. On conserve les données en DB, sans purge dans cette phase
3. Lecture directe depuis la DB
4. `traceId` :
    - nullable en base
    - mais attendu logiquement dans le flux normal
    - ne pas rigidifier la DB à ce stade
5. `assignedAgentId` :
    - normalisé avant stockage en base pour simplifier la requête
6. Stratégie d’écriture :
    - **UPDATE d’abord**
    - puis **INSERT si 0 ligne modifiée**
7. `firstSeenAt` :
    - fixé à la première insertion
    - jamais modifié ensuite

---

## 🧱 Travail à réaliser

### 1. Créer le modèle de persistance

Créer une entité JPA cohérente, par exemple :

`AdminTaskSnapshotEntity`

Table proposée :
`admin_task_snapshot`

Champs à persister :

- `taskId` (unique, not null)
- `parentTaskId` (nullable)
- `objective` (not null)
- `initiator` (not null)
- `assignedAgentId` (not null, normalisé)
- `traceId` (nullable)
- `entryPoint` (enum string, not null)
- `status` (enum string, not null)
- `channelType` (nullable)
- `conversationId` (nullable)
- `firstSeenAt` (not null)
- `lastSeenAt` (not null)
- `errorMessage` (nullable)
- `artifactCount` (not null)

Index minimaux :
- unique sur `task_id`
- index sur `last_seen_at`
- index sur `(assigned_agent_id, last_seen_at)`
- index sur `(status, last_seen_at)`
- index optionnel sur `trace_id`

⚠️ `traceId` doit être nullable en base.

---

### 2. Créer le repository

Créer un repository dédié, par exemple :

`AdminTaskSnapshotRepository`

Fonctions minimales attendues :

- recherche par `taskId`
- requêtes de lecture récente avec :
    - tri `lastSeenAt DESC`
    - filtre optionnel `assignedAgentId`
    - filtre optionnel `status`
    - limitation
- update ciblé par `taskId`

⚠️ Important :
- pas d’upsert implicite complexe
- pas de logique fragile par exception SQL comme flux principal

---

### 3. Implémenter le store persistant

Créer une implémentation DB de `AdminTaskStore`, par exemple :

`PersistentAdminTaskStore`

Elle doit devenir l’implémentation Spring principale.

⚠️ Ne pas modifier l’interface `AdminTaskStore`

---

### 4. Implémenter la méthode `record(...)`

Comportement attendu :

#### 4.1 UPDATE d’abord
Mettre à jour le snapshot existant par `taskId` :
- `status`
- `lastSeenAt`
- `errorMessage`
- `artifactCount`
- `parentTaskId`
- `objective`
- `initiator`
- `assignedAgentId`
- `traceId`
- `entryPoint`
- `channelType`
- `conversationId`

⚠️ MAIS :
- **ne jamais modifier `firstSeenAt`** dans l’UPDATE

#### 4.2 INSERT si 0 ligne modifiée
Si aucun snapshot n’existe encore :
- créer la ligne
- fixer `firstSeenAt = now`
- fixer `lastSeenAt = now`

#### 4.3 Normalisations à respecter
- `assignedAgentId` normalisé avant stockage
- `errorMessage` normalisé comme l’impl mémoire actuelle
- `artifactCount` calculé comme aujourd’hui

---

### 5. Implémenter la méthode `recent(...)`

Conserver strictement la sémantique actuelle :

- résultats triés par `lastSeenAt DESC`
- filtre optionnel sur agent
- filtre optionnel sur status
- limite appliquée

⚠️ Pour le filtre agent :
- utiliser la valeur normalisée en base
- requête simple, sans `LOWER()` en lecture si possible

---

### 6. Mapper Entity ↔ Domain

Créer un mapper propre :

- `Task + contexte` → Entity / update model
- Entity → `AdminTaskSnapshot`

⚠️ Ne pas faire de mapping inline dispersé dans le store

---

### 7. Configuration Spring

- Rendre l’impl persistante principale
- Éviter la coexistence ambiguë de deux impl actives si ce problème existe ici aussi

---

### 8. Tests

Ajouter des tests ciblés.

#### 8.1 Tests unitaires
- `record` crée un snapshot
- `record` met à jour le même `taskId` sans duplication
- `firstSeenAt` conservé
- `lastSeenAt` mis à jour
- `assignedAgentId` normalisé
- `errorMessage` normalisé
- `artifactCount` recalculé

#### 8.2 Tests intégration
- persistance réelle SQLite/JPA
- lecture `recent(...)` avec :
    - tri
    - filtre agent
    - filtre status
    - limite

#### 8.3 Test restart
- contexte A : écrire plusieurs snapshots
- contexte B sur même DB : `recent(...)` doit retrouver les données
