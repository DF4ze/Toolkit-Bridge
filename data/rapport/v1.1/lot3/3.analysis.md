# Prompt Codex — Analyse Phase 3 — Persistance des traces critiques

Tu travailles sur le projet Toolkit-Bridge / Marcel.

## 🎯 Objectif

Réaliser une **analyse ciblée** pour la Phase 3 du lot 3 :

> Persistance durable des traces critiques utiles au diagnostic

⚠️ IMPORTANT :
- Tu ne dois PAS modifier le code
- Tu ne dois PAS implémenter
- Tu dois uniquement analyser l’existant et préparer une implémentation cadrée

---

## 🧠 Contexte

Actuellement, les traces passent par :

- `DefaultAgentTraceService`
- différents `AgentTraceSink`
- notamment :
    - `InMemoryAgentTraceSink`
    - `JsonLinesAgentTraceSink`

La consultation admin lit aujourd’hui principalement la vue mémoire,
ce qui implique une perte de visibilité après redémarrage.

Le stockage JSONL existe déjà, mais il n’est pas la bonne base pour une consultation admin ciblée et simple.

---

## 📌 Décisions déjà validées

- ✔️ On ne cherche PAS à persister tout le flux d’observabilité
- ✔️ On persiste uniquement un sous-ensemble de traces critiques
- ✔️ Types retenus à ce stade :
    - `ERROR`
    - `TASK_STARTED`
    - `RESPONSE`
    - `DELEGATION`
    - `TOOL_CALL`
- ✔️ Les `attributes` sont stockés en **JSON brut**
- ❌ Pas de projection SQL avancée des attributes dans cette phase
- ❌ Pas de moteur de recherche complexe
- ❌ Pas de refonte globale de l’observabilité
- ✔️ Le JSON brut est assumé comme choix transitoire pour ne pas figer trop tôt le modèle

---

## 🔍 Travail demandé

### 1. Analyser précisément le modèle existant

Analyser au minimum :

- `AgentTraceEvent`
- `AgentTraceEventType`
- `DefaultAgentTraceService`
- `AgentTraceSink`
- `InMemoryAgentTraceSink`
- `JsonLinesAgentTraceSink`
- `TraceQueryService`
- les DTO / vues admin concernés

Pour chacun :
- rôle
- structure
- cycle de vie
- points de création
- points de lecture
- interactions

---

### 2. Cartographier le flux complet des traces

Décrire clairement :

- où les traces sont créées
- comment elles transitent
- quels sinks sont alimentés
- comment elles sont consultées côté admin
- ce qui survit au redémarrage
- ce qui est perdu

Objectif :
👉 rendre explicite la différence entre :
- trace mémoire
- trace fichier JSONL
- future trace critique persistée

---

### 3. Analyser les types d’événements retenus

Pour chacun des types suivants :

- `ERROR`
- `TASK_STARTED`
- `RESPONSE`
- `DELEGATION`
- `TOOL_CALL`

indiquer :
- utilité diagnostic / exploitation
- volume probable
- sensibilité potentielle
- risque de bruit
- valeur de persistance

Et signaler si certains de ces types nécessitent :
- filtrage complémentaire
- réduction d’attributs
- vigilance particulière

---

### 4. Analyser le champ `attributes`

Identifier :

- type réel
- exemples concrets présents dans le code
- usage réel
- volume potentiel
- risque de sensibilité
- intérêt diagnostique

Puis confirmer si le stockage JSON brut est acceptable pour cette phase,
avec les points de vigilance à documenter.

---

### 5. Proposer un modèle de persistance minimal

Sans coder, proposer :

- une table dédiée pour les traces critiques
- les colonnes à conserver
- les colonnes obligatoires / optionnelles
- les index minimaux

Le modèle doit rester :
- simple
- robuste
- compatible SQLite/JPA
- cohérent avec le besoin admin

Inclure au minimum l’analyse de :
- identifiant
- timestamp
- traceId
- agentId
- type
- message / summary si pertinent
- attributes JSON
- corrélations utiles éventuelles

⚠️ `traceId` ne doit pas être rigidifié trop tôt si l’existant ne le garantit pas toujours,
mais son rôle doit être explicitement analysé.

---

### 6. Stratégie d’intégration

Expliquer :

- comment introduire une persistance “critical traces”
- sans casser :
    - `DefaultAgentTraceService`
    - les sinks existants
    - `TraceQueryService`

Préciser si tu recommandes :
- un nouveau sink dédié DB
- ou une autre approche plus propre

Objectif :
👉 intégrer sans refactor transversal ni double logique confuse

---

### 7. Stratégie de lecture admin

Analyser comment `TraceQueryService` devrait évoluer pour lire les traces critiques persistées.

Préciser :
- si lecture DB directe recommandée
- si coexistence temporaire avec mémoire / JSONL
- quelle source doit faire foi pour la vue admin après implémentation

⚠️ Il faut éviter une ambiguïté durable entre plusieurs sources de vérité.

---

### 8. Concurrence, volume et ordre

Analyser :

- risques de concurrence
- volume potentiel
- politique minimale de tri / lecture récente
- besoin éventuel de limitation côté lecture

Sans proposer de rétention complexe à ce stade.

---

### 9. Tests à prévoir

Lister :

- tests unitaires
- tests d’intégration
- test restart

Couvrir :
- persistance d’un événement retenu
- exclusion d’un événement non retenu
- lecture triée / filtrée
- récupération après redémarrage

---

### 10. Risques et vigilance

Identifier :
- risques de sur-persistance
- risques liés au JSON brut
- risques de données sensibles
- risques de divergence entre mémoire / JSONL / DB
- risques de dette future si les attributes restent trop libres
