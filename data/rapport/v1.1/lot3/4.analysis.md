# Prompt Codex — Analyse Phase 4 — Rétention minimale

Tu travailles sur le projet Toolkit-Bridge / Marcel.

## 🎯 Objectif

Réaliser une **analyse ciblée** pour la Phase 4 du lot 3 :

> Introduire une rétention minimale sur les nouvelles données persistées du lot 3

⚠️ IMPORTANT :
- Tu ne dois PAS modifier le code
- Tu ne dois PAS implémenter
- Tu dois uniquement analyser l’existant et préparer une implémentation cadrée

---

## 🧠 Contexte

Les phases précédentes du lot 3 ont introduit une persistance durable pour :

- les interventions humaines
- les tâches admin
- les traces critiques

Nous voulons maintenant éviter une croissance infinie des données,
mais sans mettre en place un système complexe.

Le besoin est volontairement minimal :
- TTL simple
- purge planifiée
- configuration externalisée
- pas d’archivage
- pas de refonte des modèles

---

## 📌 Décisions déjà validées

### Données concernées par la rétention
- traces critiques
- tâches admin

### Données explicitement hors périmètre
- interventions humaines (pas de purge pour l’instant)

### Valeurs par défaut visées
- traces critiques → 7 jours
- tâches admin → 30 jours

### Déclenchement
- tâche planifiée (`@Scheduled`)
- fréquence configurable
- approche simple

---

## 🔍 Travail demandé

### 1. Analyser l’existant lié à la rétention

Analyser au minimum :

- repositories introduits dans les phases 2 et 3
- services de persistance concernés
- éventuelles briques de rétention déjà existantes
- configuration déjà présente dans le projet autour de la persistance / retention / cleanup

Identifier notamment :
- ce qui existe déjà
- ce qui est réutilisable
- ce qui ferait doublon
- ce qu’il vaut mieux étendre plutôt que recréer

---

### 2. Cartographier précisément les données à purger

Pour chaque type de donnée ciblé :

#### 2.1 Traces critiques
- table / repository concerné
- timestamp de référence pertinent pour la purge
- volumétrie potentielle
- impact fonctionnel d’une purge

#### 2.2 Tâches admin
- table / repository concerné
- timestamp de référence pertinent pour la purge
- impact fonctionnel d’une purge
- vigilance particulière sur `firstSeenAt` / `lastSeenAt`

#### 2.3 Interventions humaines
- expliquer pourquoi elles doivent rester hors purge dans cette phase

---

### 3. Déterminer la stratégie de purge minimale

Proposer une stratégie simple et robuste :

- critère de purge par timestamp
- fréquence d’exécution recommandée
- comportement si DB vide
- comportement si TTL absent / invalide / <= 0
- comportement si une purge échoue partiellement

Objectif :
👉 une stratégie exploitable, lisible, sans logique métier excessive

---

### 4. Proposer le modèle de configuration

Sans coder, proposer une configuration claire et minimaliste.

Par exemple, analyser la pertinence d’un bloc du type :

- `toolkit.persistence.retention.enabled`
- `toolkit.persistence.retention.cron`
- `toolkit.persistence.retention.traces-days`
- `toolkit.persistence.retention.tasks-days`

Mais avant de proposer, vérifie si le projet contient déjà une structure existante
de configuration de rétention à réutiliser ou prolonger.

⚠️ Il faut éviter de créer une seconde mécanique concurrente si quelque chose existe déjà.

---

### 5. Stratégie d’intégration

Expliquer :

- s’il faut créer un service dédié de purge
- s’il faut créer un scheduler dédié
- comment intégrer cela proprement sans coupler la purge aux services métier
- comment éviter toute ambiguïté avec une éventuelle mécanique de rétention déjà présente

---

### 6. Logging et observabilité

Proposer ce qu’il faut logguer au minimum :

- nombre de lignes supprimées
- type de données purgées
- cutoff utilisé
- anomalies éventuelles

⚠️ Pas de logs verbeux inutiles

---

### 7. Tests à prévoir

Lister :

- tests unitaires
- tests d’intégration
- cas limites utiles

Couvrir au minimum :
- purge normale
- purge avec DB vide
- TTL invalide / désactivé
- non-purge des interventions humaines
- persistance des données récentes

---

### 8. Risques et vigilance

Identifier :
- risque de doublon avec une mécanique existante
- risque de purge trop agressive
- risque de couplage à l’exécution métier
- risque de scheduler trop envahissant
- risque de dette si la rétention n’est pas cohérente avec les futures phases
