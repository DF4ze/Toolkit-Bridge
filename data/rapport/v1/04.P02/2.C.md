Tu vas implémenter la phase 2.C de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases 2.A et 2.B déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur les orchestrators déjà en place.

Objectif de cette phase :
mettre en place une architecture multi-orchestrator propre :
- résolution d’un agent vers le bon orchestrator
- factorisation des briques communes
- préparation à l’ajout futur d’autres orchestrators spécialisés

Cette phase ne doit pas encore implémenter :
- planner orchestrator
- critic orchestrator
- système avancé de sélection dynamique
- logique métier complexe dans le routeur
- couplage fort à des orchestrators futurs non encore présents

Le but est de poser un mécanisme simple, propre et extensible.

Périmètre attendu :

1. Résoudre un agent vers le bon orchestrator
   Je veux que le système sache utiliser l’orchestrator adapté selon l’agent, sa configuration ou son type d’orchestration.
   Cette résolution doit être explicite et maintenable.

2. Factoriser les briques communes
   S’il existe des éléments communs entre orchestrators, ils peuvent être factorisés.
   Mais attention :
- seulement si cela clarifie réellement l’architecture
- pas de hiérarchie abstraite inutile

3. Préparer l’ajout futur d’orchestrators spécialisés
   Je veux que l’architecture permette d’ajouter plus tard :
- `planner orchestrator`
- `critic orchestrator`
- autres orchestrators spécialisés

Sans devoir refondre le mécanisme mis en place ici.

4. Garder une séparation propre des responsabilités
   Le composant de résolution ne doit pas devenir un orchestrator déguisé.
   Chaque orchestrator garde sa responsabilité.
   Le mécanisme multi-orchestrator doit rester simple et centré sur la sélection / le dispatch.

Attendus techniques :
- créer ou adapter les classes/interfaces nécessaires
- s’appuyer proprement sur la phase 2.A et 2.B
- factoriser seulement ce qui est utile
- ajouter ou adapter les tests pertinents
- documenter brièvement les choix structurants si utile

Ce que je veux explicitement éviter :
- routeur surchargé de logique métier
- abstraction générique vide
- architecture trop théorique
- dépendances circulaires entre orchestrators
- anticipation excessive des orchestrators futurs

Livrables attendus :
1. code complet de la phase 2.C
2. tests
3. résumé court des choix autour du multi-orchestrator
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les futures phases

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
