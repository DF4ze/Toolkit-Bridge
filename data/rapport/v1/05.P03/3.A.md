Tu vas implémenter la phase 3.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le runtime agentique et les orchestrators existants.

Objectif de cette phase :
introduire un `TaskModel` minimal, propre et extensible, servant de base aux futures exécutions orientées objectif et aux sous-tâches.

Cette phase ne doit pas encore implémenter :
- un bus inter-agent complet
- une délégation distribuée réelle
- une orchestration complexe de sous-tâches
- un moteur complet de workflow
- un dashboard
- une logique avancée de traces

Le but est de poser une entité `Task` claire, stable et réutilisable.

Périmètre attendu :

1. Introduire une entité `Task` minimale
   Je veux définir une entité métier de base avec au minimum :
- `taskId`
- objectif
- initiateur
- agent assigné
- `parentTaskId`
- metadata

Cette entité doit être simple, lisible et conçue pour évoluer sans refactor lourde.

2. Prévoir les artefacts liés
   Je veux que la tâche puisse porter ou référencer une liste d’artefacts liés.
   Attention :
- ne pas sur-implémenter tout le système d’artefacts ici si la phase dédiée n’est pas encore faite
- mais prévoir proprement le point d’accroche nécessaire

3. Préparer la corrélation avec les traces
   La tâche doit pouvoir être corrélée plus tard avec les traces d’exécution.
   Il faut donc penser le modèle de façon compatible avec une future observabilité structurée, sans implémenter ici toute la mécanique de tracing.

4. Clarifier le point d’entrée métier
   Je veux que le code prépare explicitement les règles suivantes :
- une conversation utilisateur peut rester dans l’orchestrator `chat`
- une exécution agentique orientée objectif passe par l’orchestrator `task`
- une délégation devra créer une sous-tâche

Attention :
- il ne s’agit pas forcément d’implémenter ici toute la délégation
- mais le modèle ne doit pas empêcher cette évolution

5. Garder la séparation avec l’orchestrator
   Le `TaskModel` ne doit pas être noyé dans la logique d’orchestration.
   Je veux un vrai modèle métier, pas un simple DTO opportuniste caché dans un service.

Attendus techniques :
- créer ou adapter les classes/interfaces nécessaires
- garder les responsabilités nettes
- rester cohérent avec les phases précédentes
- éviter les dépendances inutiles
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- entité Task surchargée dès la V1
- mélange entre modèle métier et logique d’exécution
- dépendance forte au bus ou à la délégation alors qu’ils ne sont pas encore finalisés
- simulation artificielle de futures features
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 3.A
2. tests
3. résumé court des choix autour du `TaskModel`
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 3.B et les phases suivantes

- puis implémente
