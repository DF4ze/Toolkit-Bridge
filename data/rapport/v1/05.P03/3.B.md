Tu vas implémenter la phase 3.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 3.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le `TaskModel` déjà introduit.

Objectif de cette phase :
définir le cycle de vie V1 des tâches et encadrer les transitions d’état de façon simple, cohérente et robuste.

Cette phase ne doit pas encore implémenter :
- un moteur complet de workflow
- une orchestration distribuée multi-agent avancée
- des règles complexes de reprise automatique
- un dashboard
- une logique métier de supervision avancée

Le but est de poser une machine de cycle de vie simple, utile et fiable.

Périmètre attendu :

1. Introduire les statuts V1
   Je veux au minimum les statuts suivants :
- `CREATED`
- `RUNNING`
- `WAITING`
- `DONE`
- `FAILED`
- `CANCELLED`

Le modèle doit rester simple mais exploitable, et préparer un futur dashboard.

2. Encadrer les transitions
   Je veux définir les transitions légitimes entre statuts.
   Le système doit empêcher ou bloquer les changements incohérents.

Attention :
- éviter une machine à états inutilement lourde
- mais ne pas laisser les transitions libres et implicites partout dans le code

3. Garder la logique de transition au bon endroit
   La validation des transitions ne doit pas être dispersée dans plusieurs services.
   Je veux une logique claire, centralisée et testable.

4. Préparer audit et debug
   Le cycle de vie doit aider à :
- comprendre où en est une tâche
- détecter une transition invalide
- préparer de futures traces ou vues de supervision

Sans introduire maintenant toute l’observabilité complète.

5. Rester compatible avec les futures sous-tâches
   La mécanique mise en place ici doit être compatible avec :
- des sous-tâches
- des attentes intermédiaires
- une future orchestration plus riche

Mais sans implémentation prématurée.

Attendus techniques :
- créer ou adapter les classes nécessaires
- intégrer proprement la logique de statut au `TaskModel` ou à un composant dédié, selon ce qui est le plus sain
- ajouter ou adapter les tests pertinents
- documenter brièvement les choix structurants si utile

Ce que je veux explicitement éviter :
- transitions validées de manière dispersée
- machine à états trop compliquée
- statuts inutiles en V1
- logique de workflow avancée déguisée
- couplage fort avec des phases futures non encore implémentées

Livrables attendus :
1. code complet de la phase 3.B
2. tests
3. résumé court des choix autour du cycle de vie des tâches
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
