Tu vas implémenter la phase 5.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le modèle de tâche, le bus inter-agent et le runtime multi-agent.

Objectif de cette phase :
introduire un vrai type métier `Artifact`, clair, typé et réutilisable, afin de sortir du tout-prompt textuel et de structurer les productions et échanges du système.

Cette phase ne doit pas encore implémenter complètement :
- un système de stockage complet des artefacts
- une rétention complète
- une persistance lourde ou définitive
- un système de versioning avancé
- une logique de partage distribué complexe

Le but est de poser le modèle métier et ses points d’intégration.

Périmètre attendu :

1. Introduire un type métier `Artifact`
   Je veux un vrai modèle d’artefact, distinct des tâches, messages et prompts.
   Ce modèle doit être réutilisable dans les futures phases.

2. Définir les familles V1
   Je veux au minimum les familles suivantes :
- report
- patch
- script
- plan
- summary
- file
- memory candidate

Attention :
- le typage doit être clair et exploitable
- éviter les strings magiques dispersées
- garder un modèle simple et extensible

3. Relier artefacts et tâches
   Je veux qu’un artefact puisse être associé à une tâche donnée.
   L’objectif est de :
- rattacher les sorties à leur contexte d’exécution
- préparer reprise, audit et partage inter-agent
- éviter que le résultat d’une tâche soit seulement du texte libre

4. Préparer l’usage dans les messages
   Le modèle doit être compatible avec les échanges inter-agent.
   Il faut donc prévoir que certains messages puissent référencer ou transporter des artefacts, sans implémenter maintenant une mécanique trop lourde.

5. Garder la séparation des responsabilités
   Je veux éviter :
- qu’un artefact soit juste un champ annexe mal défini dans `Task`
- que la structure d’artefact soit dictée uniquement par le stockage futur
- que le modèle métier soit noyé dans les logs ou les messages

Attendus techniques :
- créer ou adapter les classes/interfaces nécessaires
- garder des noms de classes en anglais
- rester cohérent avec le modèle de tâche et le bus existants
- éviter les dépendances inutiles
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- artefact purement opportuniste
- mélange entre modèle métier et stratégie de stockage
- sur-ingénierie
- dépendance forte à une future persistance non encore stabilisée
- duplication évitable entre tâches, messages et artefacts

Livrables attendus :
1. code complet de la phase 5.A
2. tests
3. résumé court des choix autour du modèle `Artifact`
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 5.B

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
