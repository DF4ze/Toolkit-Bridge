Tu vas implémenter la phase 5.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 5.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le modèle `Artifact` déjà introduit.

Objectif de cette phase :
gérer proprement le stockage des artefacts en séparant métadonnées et contenu, et préparer une rétention pilotable par type.

Cette phase ne doit pas encore implémenter :
- un système d’archivage complet
- une stratégie distribuée de stockage
- une UI d’administration
- une gouvernance complète du cycle de vie des artefacts
- un versioning avancé

Le but est de poser une base simple, claire et maintenable pour le stockage et la rétention.

Périmètre attendu :

1. Séparer métadonnées et contenu
   Je veux éviter de tout mélanger.
   L’architecture doit permettre de distinguer :
- les métadonnées structurées de l’artefact
- le contenu ou payload stocké dans le workspace ou un stockage dédié

Attention :
- ne pas figer inutilement une architecture trop complexe
- mais poser une vraie séparation de responsabilité

2. Définir un mode de stockage cohérent
   Je veux une base saine pour le stockage des artefacts :
- métadonnées en modèle structuré ou persistance adaptée
- contenu dans workspace ou stockage dédié selon ce qui est le plus cohérent avec le projet actuel

Le choix doit rester simple et compatible avec l’existant.

3. Prévoir la rétention par type
   Je veux préparer des règles distinctes selon la nature de l’artefact.
   L’objectif est de faciliter :
- nettoyage
- archivage futur
- pilotage par configuration

Attention :
- ne pas disperser la logique de rétention partout
- ne pas tout archiver ou tout supprimer de manière uniforme

4. Garder le stockage indépendant des logs
   Je veux explicitement éviter que les artefacts soient essentiellement stockés ou reconstruits à partir des logs.
   Les logs et traces peuvent référencer un artefact, mais ne doivent pas en devenir le stockage principal.

5. Préparer les futures évolutions sans les implémenter
   Cette phase doit préparer plus tard :
- archivage
- purge
- administration
- consultation par dashboard ou UI

Mais sans les réaliser maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services nécessaires
- garder les responsabilités nettes entre modèle, stockage et rétention
- rester cohérent avec workspace, tasks et messages
- rendre la rétention pilotable par configuration si pertinent dès cette phase
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- stockage trop couplé à un seul cas d’usage
- rétention codée en dur dans plusieurs endroits
- confusion entre métadonnées et contenu
- artefacts noyés dans les logs
- sur-conception

Livrables attendus :
1. code complet de la phase 5.B
2. tests
3. résumé court des choix autour du stockage et de la rétention des artefacts
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
