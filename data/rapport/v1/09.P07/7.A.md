Tu vas implémenter la phase 7.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment les briques mémoire, le `ContextAssembler` et le système de workspace.

Objectif de cette phase :
introduire un support de contexte global partagé, simple, humainement éditable et rapidement exploitable par les agents, au moyen d’un ou plusieurs fichiers Markdown de référence.

Cette phase ne doit pas encore implémenter :
- une grosse base dédiée pour le contexte global
- une UI d’administration
- des tools mémoire explicites avancés
- une gouvernance complète des droits d’écriture
- une synchronisation complexe entre plusieurs sources

Le but est de poser une source de contexte global simple, claire et maintenable.

Périmètre attendu :

1. Introduire un support de contexte global partagé
   Je veux que le système puisse charger un ou plusieurs fichiers Markdown de référence représentant un contexte global partagé.

Exemples de contenu visé :
- profil utilisateur
- règles globales
- contexte de projets
- conventions durables

L’objectif est d’avoir une solution simple à éditer humainement, sans base dédiée lourde.

2. Utiliser un ou plusieurs fichiers Markdown de référence
   Le système doit permettre une structure simple et explicite.
   Tu peux t’appuyer sur un ou plusieurs fichiers, selon ce qui est le plus propre dans l’architecture actuelle.

Attention :
- ne pas transformer cela en mini-CMS
- ne pas disperser les conventions de chargement un peu partout
- garder un format lisible et éditable

3. Prévoir deux modes de chargement
   Je veux supporter deux modes :
- lecture à la volée lors de l’assemblage du contexte
- ou cache mémoire avec rechargement contrôlé

Le choix du mode doit être pilotable via properties.

4. Intégrer proprement avec le `ContextAssembler`
   Le contexte global partagé doit pouvoir être injecté dans le contexte LLM via la mécanique déjà en place.
   Attention :
- ne pas dupliquer la logique de lecture dans plusieurs composants
- garder une responsabilité claire entre lecture/chargement et assemblage de contexte

5. Préparer les évolutions futures sans les implémenter
   Cette phase doit préparer plus tard :
- des tools de lecture/écriture du contexte global
- des règles d’écriture plus fines
- une administration plus conviviale

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/properties nécessaires
- garder les responsabilités nettes
- rester cohérent avec le système de mémoire et de workspace existant
- rendre le comportement lisible, configurable et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- stockage du contexte global codé en dur
- logique de chargement dispersée
- confusion entre contexte global et conversation memory
- architecture trop lourde pour un simple besoin de base
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 7.A
2. tests
3. résumé court des choix autour du support de contexte global partagé
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 7.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
