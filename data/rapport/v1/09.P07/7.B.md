Tu vas implémenter la phase 7.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 7.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le support de contexte global partagé déjà introduit ainsi que sur le système de workspace existant.

Objectif de cette phase :
formaliser clairement la frontière entre le contexte global partagé et le shared workspace, afin d’éviter de mélanger connaissances stables et travail courant.

Cette phase ne doit pas encore implémenter :
- une politique complète de permissions fines
- une UI d’administration
- un système avancé de validation des écritures
- une gouvernance complète multi-agent du shared workspace

Le but est de poser des frontières saines et explicites dans l’architecture.

Périmètre attendu :

1. Formaliser le rôle du contexte global partagé
   Je veux réserver le contexte global partagé aux connaissances stables et relisibles par les agents.

Exemples :
- nom utilisateur
- règles durables
- conventions
- contexte projet stable

Ces fichiers doivent être pensés pour être relus régulièrement par les agents et rester lisibles/éditables.

2. Formaliser le rôle du shared workspace
   Je veux réserver le shared workspace au travail courant.

Exemples :
- rapports
- scripts
- exports
- artefacts
- brouillons

Il faut éviter d’y mélanger les règles globales ou le contexte stable.

3. Poser des règles explicites de frontière
   Je veux que l’architecture et/ou la configuration rendent claire la différence entre :
- ce qui constitue une source de contexte stable
- ce qui constitue une zone de production ou d’échange courant

Attention :
- ne pas laisser cette frontière comme une convention implicite dans la tête du développeur
- la rendre visible dans le code, les types, la config ou les services selon ce qui est le plus sain

4. Mieux contrôler qui écrit quoi
   Sans implémenter une gouvernance complète, cette phase doit préparer une meilleure maîtrise de l’écriture :
- éviter que n’importe quel flux écrive n’importe où
- préparer les futures policies et permissions
- clarifier les points d’entrée d’écriture

5. Préparer les évolutions futures sans les implémenter
   Cette phase doit préparer plus tard :
- tools mémoire explicites
- validation d’écriture
- administration
- politiques d’accès plus fines

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/properties nécessaires
- clarifier la structure et les responsabilités
- rendre la frontière visible et maintenable
- rester cohérent avec les phases précédentes
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- mélange entre contexte global et shared workspace
- conventions implicites non matérialisées
- architecture trop théorique
- permissions fines prématurées
- sur-conception

Livrables attendus :
1. code complet de la phase 7.B
2. tests
3. résumé court des choix autour de la frontière entre contexte global et shared workspace
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
