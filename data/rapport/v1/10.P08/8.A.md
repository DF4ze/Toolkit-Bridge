Tu vas implémenter la phase 8.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment les briques mémoire, les scopes mémoire, le `ContextAssembler`, le système de tools et les policies.

Objectif de cette phase :
introduire des tools mémoire explicites V1, simples, utiles et bien cadrés, afin de permettre un accès explicite en lecture/écriture aux mémoires réellement utiles.

Cette phase ne doit pas encore implémenter :
- un grand système de gestion mémoire complet
- des tools mémoire très nombreux
- une gouvernance complète des validations
- un pipeline d’écriture implicite complexe
- une UI d’administration mémoire
- un système avancé de fusion, versioning ou modération mémoire

Le but est de poser un petit socle de tools mémoire explicites, cohérent avec l’architecture actuelle.

Périmètre attendu :

1. Introduire des tools mémoire explicites V1
   Je veux un premier ensemble réduit de tools mémoire permettant un accès explicite et utile aux mémoires importantes.

2. Prévoir un accès RW sur les mémoires utiles
   Je veux couvrir au minimum les usages suivants :
- lire un contexte mémoire
- ajouter ou modifier un fait
- ajouter ou modifier une règle

Attention :
- rester cohérent avec les scopes mémoire et le modèle métier déjà introduits
- ne pas permettre un accès flou ou fourre-tout à “toute la mémoire”
- garder des points d’entrée clairs et typés

3. Commencer petit
   Je veux explicitement commencer avec un petit socle V1, par exemple autour de :
- un tool de rappel de contexte
- un tool d’écriture d’un fait
- un tool d’écriture d’une règle

Attention :
- ne pas multiplier les tools prématurément
- privilégier la clarté métier à l’exhaustivité

4. Garder une vraie intention métier dans les tools
   Je veux que les tools reflètent des usages explicites et non ambigus.
   Par exemple :
- écrire un fait n’est pas écrire une règle
- lire un contexte n’est pas faire une fouille mémoire générique sans cadre

5. Respecter les frontières d’architecture déjà posées
   Les tools mémoire doivent s’appuyer proprement sur les services/modèles mémoire existants.
   Je veux éviter :
- logique mémoire codée directement dans le tool handler
- duplication de logique avec le `ContextAssembler`
- accès non contrôlé qui contourne les policies futures

6. Préparer la suite sans la sur-implémenter
   Cette phase doit préparer plus tard :
- enrichissement des tools mémoire
- contrôle plus fin des écritures
- outils de correction/suppression
- administration mémoire

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/tools nécessaires
- garder les responsabilités nettes entre tools, services mémoire et modèles métier
- rester cohérent avec les scopes mémoire et le système de tools existant
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- tool mémoire générique trop flou
- mélange entre fait, règle et contexte projet
- logique métier dispersée dans les handlers
- contournement implicite des futures policies
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 8.A
2. tests
3. résumé court des choix autour des tools mémoire V1
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 8.B et les phases suivantes
