Tu vas implémenter la phase 8.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 8.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur les briques mémoire et les tools mémoire explicites déjà en place.

Objectif de cette phase :
distinguer proprement l’écriture mémoire explicite de l’écriture mémoire implicite, afin de réduire les ambiguïtés métier et de poser des types d’intention clairs.

Cette phase ne doit pas encore implémenter :
- un pipeline d’extraction implicite très sophistiqué
- une classification LLM avancée
- un système de modération ou validation complexe
- une gouvernance complète des conflits mémoire
- une UI de gestion mémoire

Le but est de poser une séparation claire entre deux modes d’écriture mémoire :
- explicite
- implicite

Périmètre attendu :

1. Distinguer explicite et implicite
   Je veux que l’architecture rende claire la différence entre :
- une écriture explicite demandée volontairement
- une écriture implicite issue d’une extraction automatique après échange ou exécution

Cette distinction doit être visible dans les modèles, services, types ou pipelines selon ce qui est le plus sain.

2. Garder une écriture automatique séparée
   Je veux qu’un pipeline d’écriture implicite reste distinct des tools ou commandes explicites.
   L’objectif est de :
- éviter les ambiguïtés métier
- éviter qu’une commande explicite déclenche le même traitement qu’une extraction automatique
- garder un système plus lisible et mieux gouvernable

Attention :
- ne pas implémenter ici un pipeline implicite trop ambitieux
- poser surtout la séparation structurelle

3. Donner une vraie intention d’écriture
   Je veux des types métier clairs.
   Par exemple :
- une règle explicite ne doit pas être traitée comme un simple fait
- un contexte projet ne doit pas être noyé dans la conversation
- l’origine et l’intention d’une écriture doivent rester compréhensibles

4. Préparer la suite sans la figer
   Cette phase doit préparer plus tard :
- une meilleure extraction implicite
- des politiques de validation
- des stratégies de priorité ou de confiance
- des outils d’édition/correction plus riches

Mais sans les implémenter maintenant.

5. Garder la compatibilité avec le système existant
   Je veux éviter une refonte brutale des briques mémoire déjà en place.
   Si une adaptation est nécessaire, elle doit rester structurante, simple et justifiée.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/types nécessaires
- clarifier la séparation entre pipeline explicite et pipeline implicite
- poser des types métier ou marqueurs clairs si pertinent
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- écriture mémoire implicite et explicite mélangées
- fait/règle/contexte projet traités comme de simples textes sans intention
- pipeline automatique trop ambitieux à ce stade
- architecture trop théorique
- sur-conception

Livrables attendus :
1. code complet de la phase 8.B
2. tests
3. résumé court des choix autour de la séparation explicite/implicite
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
