Après relecture :
Faiblesses

Le service applicatif des tools mémoire dépendait du contexte d’authentification runtime via CurrentAgentService, ce qui créait un couplage caché entre couche applicative mémoire et infrastructure d’exécution. C’était le point le plus gênant pour les futures phases, car le service n’était plus réutilisable hors exécution tool. Corrigé dans ExplicitMemoryToolService.java (line 33) et reporté aux handlers runtime RecallMemoryContextToolHandler.java (line 18), WriteMemoryFactToolHandler.java (line 22), WriteMemoryRuleToolHandler.java (line 20).

Le tool de rappel de contexte réutilisait bien le pipeline de contexte, mais exposait aussi la section ## User Input injectée avec un texte artificiel de rappel. C’était une fuite d’un détail de runtime LLM dans un use case de lecture mémoire. Corrigé dans ExplicitMemoryToolService.java (line 50).

Le modèle writeMode est utile et cohérent, mais il reste aujourd’hui faiblement garanti côté persistance pour préserver la compatibilité de migration: les colonnes sont tolérantes à null et la valeur est surtout sécurisée par défaut applicatif. Ce n’est pas bloquant maintenant, mais c’est une dette à refermer plus tard avec une migration/backfill propre. Voir MemoryEntry.java (line 88) et RuleEntry.java (line 70).

Le découplage orchestrator / mémoire / tooling a progressé, mais la gouvernance outillée reste encore globalement binaire: ces nouveaux tools sont bien séparés métier, sans pour autant être encore gouvernés par une policy fine par tool. Ce n’est pas une régression introduite ici, plutôt une limite structurelle existante à garder en tête.

Les handlers restent lisibles, mais ils dupliquent un peu de parsing d’arguments (asString, asLong, etc.). Je n’ai pas factorisé cela maintenant, car ce serait surtout cosmétique et risquerait d’ajouter une abstraction prématurée.

Corrections utiles proposées

Sortir la dépendance runtime du service applicatif explicite.
Empêcher le tool de rappel de contexte d’exposer un faux User Input.
Renforcer les tests pour vérifier que l’agentId runtime est bien injecté par les handlers.
Ne pas toucher au reste tant qu’on n’a pas une vraie phase dédiée à la gouvernance fine des tools et aux migrations de schéma.
Corrections appliquées

Le service explicite mémoire ne dépend plus de CurrentAgentService; l’agentId est désormais porté explicitement par les requêtes métier et fourni par les handlers runtime.
Le rappel de contexte nettoie la section synthétique ## User Input avant de retourner le contexte.
Les tests couvrent maintenant explicitement la propagation de l’agentId par les handlers et le nettoyage du contexte rappelé.
Je n’ai pas ajouté de nouvelle fonctionnalité hors périmètre.
Résumé final
L’architecture est globalement saine pour la phase: la séparation explicite/implicite est visible, les tools V1 ont une intention métier claire, et le MemoryFacade n’est plus chargé de la logique implicite détaillée grâce à DefaultMemoryFacade.java (line 34) et ImplicitMemoryWritePipeline.java (line 18). Le point le plus discutable a été corrigé: la couche applicative mémoire n’a plus de dépendance cachée au runtime d’authentification.

Les principales dettes restantes sont raisonnables et assumables pour cette phase: writeMode pas encore durci au niveau DB, gouvernance par tool encore grossière, et un peu de parsing dupliqué dans les handlers. Les tests ajoutés sont utiles et ciblés, mais je n’ai toujours pas pu faire tourner une validation Maven complète, car le build reste bloqué par l’erreur préexistante sur DefaultController.java.
