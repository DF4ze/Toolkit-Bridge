Tu vas implémenter la phase 11.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le système de tools existant, les policies par agent, le runtime multi-agent et les futures évolutions prévues autour des scripted tools.

Objectif de cette phase :
catégoriser officiellement les tools de la plateforme, afin de distinguer clairement les familles de tools et de leur associer un niveau de risque exploitable par la policy et la gouvernance future.

Cette phase ne doit pas encore implémenter :
- un système complet de scripted tools
- une validation humaine réelle
- une gouvernance avancée des approbations
- un registre complet de tools enrichi
- une UI d’administration des tools

Le but est de poser une taxonomie propre, simple et durable.

Périmètre attendu :

1. Distinguer officiellement `native` et `scripted`
   Je veux que la plateforme distingue explicitement au moins deux familles de tools :
- `native` : robustes, testables, intégrés au code
- `scripted` : plus souples, plus encadrés, pensés pour évoluer plus tard

Attention :
- ce ne doit pas être une simple convention implicite
- cette distinction doit être visible dans les types, le modèle ou les contrats pertinents
- éviter les strings magiques dispersées

2. Associer un niveau de risque à chaque tool
   Je veux qu’un tool puisse être associé à un niveau de risque ou une classe de risque claire, par exemple autour de :
- lecture seule
- écriture locale
- exécution
- sortie externe

L’objectif est d’aider :
- la policy
- la validation future
- la gouvernance

Attention :
- rester simple en V1
- ne pas construire une taxonomie inutilement complexe

3. Garder des contrats clairs pour chaque catégorie
   Je veux que la différence entre catégories de tools soit exploitable dans l’architecture.
   Il faut éviter :
- des catégories purement décoratives
- des responsabilités floues
- des distinctions non utilisées ou trop implicites

4. Préparer les futures extensions sans les implémenter
   Cette phase doit préparer plus tard :
- le support réel des scripted tools
- des règles de validation selon le risque
- une gouvernance plus fine
- des contrôles plus riches dans la policy

Mais sans les implémenter maintenant.

5. Rester cohérent avec l’existant
   Je veux éviter une refonte brutale des tools déjà présents.
   Si une adaptation est nécessaire, elle doit rester structurante, simple et justifiée.

Attendus techniques :
- créer ou adapter les classes/interfaces/enums/modèles nécessaires
- rendre la catégorisation et le risque visibles et utilisables
- rester cohérent avec la policy et le système de tools existant
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- catégories décoratives
- niveau de risque trop fin ou trop théorique en V1
- logique de gouvernance prématurée
- rupture inutile avec les tools existants
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 11.A
2. tests
3. résumé court des choix autour des catégories et niveaux de risque des tools
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 11.B et les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
