Tu vas implémenter la phase 13.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 13.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le système de traces agentiques déjà introduit.

Objectif de cette phase :
définir les événements clés à tracer dans le flux agentique et préparer plusieurs destinations d’exploitation, sans construire encore tout le système de supervision final.

Cette phase ne doit pas encore implémenter :
- un dashboard complet
- une UI d’administration
- une supervision Telegram complète
- une analyse avancée en temps réel
- une stack de collecte externe complexe

Le but est de rendre les traces réellement utiles dans le flux métier courant.

Périmètre attendu :

1. Définir les événements clés
   Je veux couvrir au minimum les étapes utiles du flux agentique, selon ce qui existe déjà réellement dans le projet :
- début de tâche
- assemblage de contexte
- tool call
- délégation
- réponse
- erreur

Attention :
- ne pas inventer artificiellement des événements pour des phases non encore réellement implémentées
- rester cohérent avec l’état du dépôt

2. Donner une vision chronologique claire
   Je veux que les traces permettent de reconstruire une chronologie métier lisible.
   L’objectif est qu’un humain ou un futur système de supervision puisse comprendre ce qui s’est passé.

3. Préparer plusieurs destinations
   Je veux préparer au moins une architecture compatible avec plusieurs destinations possibles, par exemple :
- fichier JSON par agent
- flux centralisé
- autre point d’exploitation futur

Attention :
- ne pas sur-implémenter un pipeline complexe
- garder une abstraction simple et maintenable

4. Préparer l’exploitation future
   Cette phase doit servir de base saine pour plus tard :
- dashboard
- UI d’administration
- supervision
- méta-analyse

Mais sans les implémenter maintenant.

5. Garder une séparation claire des responsabilités
   Je veux éviter que :
- chaque composant métier décide lui-même du format final des traces
- les destinations soient codées en dur partout
- le système de traces devienne une logique métier centrale confuse

Attendus techniques :
- créer ou adapter les classes/interfaces/services nécessaires
- définir les événements clés de façon explicite et maintenable
- brancher proprement les points d’émission pertinents existants
- garder les destinations interchangeables si pertinent
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- événements trop flous
- émissions de traces dispersées sans cohérence
- destinations codées en dur partout
- anticipation excessive de la supervision finale
- sur-conception

Livrables attendus :
1. code complet de la phase 13.B
2. tests
3. résumé court des choix autour des événements clés et des destinations de traces
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
