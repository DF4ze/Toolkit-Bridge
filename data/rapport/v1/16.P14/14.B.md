Tu vas implémenter la phase 14.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 14.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’intégration Telegram de supervision déjà introduite, ainsi que sur le bus inter-agent et les traces agentiques.

Objectif de cette phase :
préparer l’intervention humaine future via Telegram, sans faire de Telegram le cœur du système et sans implémenter encore de contrôle actif complet.

Cette phase ne doit pas encore implémenter :
- une validation humaine opérationnelle de bout en bout
- un blocage réel en production piloté depuis Telegram
- un workflow complet d’approbation
- une logique métier dépendante du canal Telegram

Le but est de poser les bons points d’extension pour une future validation humaine légère.

Périmètre attendu :

1. Envisager un mode validation plus tard
   Je veux que l’architecture ouvre la voie à un futur scénario où un humain pourra :
- approuver une action
- bloquer une action
- utiliser Telegram comme interface légère

Attention :
- ne pas implémenter ici tout le workflow final
- poser seulement les abstractions, états ou points d’extension réellement utiles

2. Ne pas faire de Telegram le cœur du système
   Je veux que le bus interne reste la source de vérité.
   Telegram doit rester une exposition ou une interface périphérique.
   Le métier doit rester indépendant du canal.

3. Préparer une future interaction humaine sans casser le découplage
   Je veux éviter :
- que des handlers Telegram contiennent la logique métier profonde
- que la validation soit pensée uniquement pour Telegram
- que le bus dépende du retour Telegram pour fonctionner dès maintenant

L’idée est d’ouvrir la voie, pas de figer le système autour de Telegram.

4. Garder la mise en place légère et non intrusive
   Cette phase doit rester préparatoire.
   Elle doit surtout rendre possible plus tard :
- validation humaine
- blocage ciblé
- approbation légère
- interaction ponctuelle depuis le groupe

Mais sans basculer trop tôt dans un workflow complexe.

5. Rester cohérent avec la policy et la gouvernance future
   La préparation faite ici doit être compatible avec :
- policies
- validation future de scripted tools
- actions sensibles
- supervision

Mais sans réimplémenter ces briques ici.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/types nécessaires
- préparer les bons points d’extension sans couplage fort
- garder les responsabilités nettes entre bus, métier, policy et canal Telegram
- rendre le code lisible et testable
- ajouter ou adapter les tests pertinents si possible

Ce que je veux explicitement éviter :
- workflow complet prématuré
- dépendance métier à Telegram
- logique d’approbation codée en dur dans le bot Telegram
- architecture trop théorique
- sur-conception

Livrables attendus :
1. code complet de la phase 14.B
2. tests
3. résumé court des choix autour de la préparation de l’intervention humaine via Telegram
