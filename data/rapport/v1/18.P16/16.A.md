Tu vas implémenter la phase 16.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment le système de workspace, les artefacts, les traces agentiques, la plateforme de tools et la supervision existante.

Objectif de cette phase :
sortir certains process du code dur en les stockant dans des fichiers modifiables, versionnables de manière simple, sauvegardables et auditables, afin de réduire le risque porté par le code Java et préparer une future amélioration pilotée par agent méta.

Cette phase ne doit pas encore implémenter :
- un agent méta complet
- une auto-modification autonome
- un système complet de versioning avancé
- une UI d’administration dédiée
- un workflow complexe de revue/approbation

Le but est de poser un support externe de process, simple, maintenable et sûr.

Périmètre attendu :

1. Sortir les process du code dur
   Je veux que certains process puissent être représentés dans des fichiers modifiables plutôt que figés uniquement dans le code Java.

L’objectif est de :
- rendre ces process plus faciles à faire évoluer
- les rendre auditables
- réduire le risque de modification directe du code pour des ajustements de process

2. Stocker les process dans des fichiers modifiables
   Je veux un support de fichiers adapté, simple et cohérent avec l’architecture actuelle.
   Ces fichiers doivent être :
- modifiables
- sauvegardables
- auditables
- exploitables par le système plus tard

Attention :
- ne pas créer un mini langage exotique inutile
- privilégier un format lisible, simple et versionnable
- rester cohérent avec le shared workspace ou un emplacement dédié si c’est plus sain

3. Prévoir le support d’un futur agent méta
   Cette phase doit préparer plus tard l’intervention d’un agent dédié qui pourra analyser ou proposer des modifications de process.
   Mais il ne faut pas implémenter cet agent maintenant.

4. Prévoir un backup avant modification
   Je veux qu’avant toute réécriture d’un process, une sauvegarde automatique soit réalisée.
   Je veux aussi préparer :
- un historique simple des changements
- un retour arrière possible

Attention :
- rester simple en V1
- ne pas construire un système complet de gestion documentaire ou de versioning avancé

5. Garder les responsabilités propres
   Je veux éviter :
- que la logique de backup soit dispersée
- que les process externes soient mélangés avec des artefacts jetables
- que l’accès en écriture soit implicite ou incontrôlé

6. Préparer la suite sans la sur-implémenter
   Cette phase doit préparer plus tard :
- revue de process
- génération de nouvelles versions
- approbation
- exploitation par un agent méta

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/properties nécessaires
- définir un emplacement cohérent pour les process externes
- séparer proprement contenu, sauvegarde et historique simple
- rester cohérent avec workspace, artefacts et traces
- garder le code lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- mini moteur de workflow complet
- système de versioning lourd
- process stockés de façon floue
- backup bricolé dans plusieurs endroits
- sur-ingénierie

Livrables attendus :
1. code complet de la phase 16.A
2. tests
3. résumé court des choix autour des process externes modifiables
4. liste des fichiers modifiés/créés
5. points laissés volontairement ouverts pour la phase 16.B et les phases suivantes

- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
