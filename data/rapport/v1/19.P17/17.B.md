Tu vas implémenter la phase 17.B de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 17.A déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le typage des objets persistables déjà introduit, ainsi que sur les mécanismes de stockage existants.

Objectif de cette phase :
appliquer des règles de rétention par type d’objet, configurables et compréhensibles, afin de préparer proprement purge, archivage et conservation utile de l’historique.

Cette phase ne doit pas encore implémenter :
- un système complet d’archivage long terme
- une purge distribuée complexe
- une UI complète de pilotage de rétention
- des règles réglementaires avancées ou contextuelles très fines

Le but est de poser une base centralisée, pilotable et durable pour la rétention.

Périmètre attendu :

1. Rendre la durée configurable
   Je veux que les règles de rétention soient pilotables par configuration, idéalement via properties ou un mécanisme équivalent cohérent avec le projet.

L’objectif est que la durée de conservation puisse être ajustée selon :
- le type d’objet
- le domaine métier
- des besoins futurs plus stricts

2. Appliquer des règles par type
   Je veux que la rétention soit pensée par famille d’objet persistable, et non comme une règle uniforme pour tout le système.

Attention :
- ne pas disperser les règles dans plusieurs composants
- garder un point de pilotage clair et compréhensible

3. Préparer archivage et purge
   Je veux que l’architecture introduite ici prépare proprement la distinction suivante :
- purger ce qui est jetable
- archiver ce qui est critique
- préserver l’historique quand il a de la valeur

Attention :
- ne pas implémenter ici un archivage très riche si le projet n’en a pas encore besoin
- poser surtout les bonnes frontières et les bons points d’extension

4. Garder la logique de rétention centralisée
   Je veux éviter :
- de la rétention codée en dur un peu partout
- des décisions implicites dans chaque service
- des règles différentes selon les développeurs

La stratégie doit rester lisible, maintenable et testable.

5. Rester cohérent avec les futures phases d’administration
   Cette phase doit préparer plus tard :
- pilotage via UI d’admin
- supervision de la rétention
- archivage plus fin
- nettoyage plus avancé

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/properties nécessaires
- brancher proprement les règles de rétention sur les familles persistables existantes
- garder les responsabilités nettes entre typage, stockage, archivage futur et purge
- rendre le système lisible et testable
- ajouter ou adapter les tests pertinents

Ce que je veux explicitement éviter :
- durée codée en dur dans plusieurs endroits
- règle unique pour tous les objets
- purge/archivage prématurément trop complexes
- architecture confuse
- sur-conception

Livrables attendus :
1. code complet de la phase 17.B
2. tests
3. résumé court des choix autour des règles de rétention par type
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les phases suivantes

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
