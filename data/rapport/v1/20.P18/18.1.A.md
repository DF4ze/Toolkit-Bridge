Tu vas travailler sur la phase 18.1.A de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet et la logique d’implémentation incrémentale te sont fournis séparément.
Tu dois t’y conformer strictement.

Objectif de cette phase :
réaliser une cartographie claire de la configuration actuelle du projet, classifier les paramètres selon leur nature et leur source cible, et identifier les accès qui devront être refactorés avant l’administration réelle.

Important :
cette phase est d’abord une phase d’analyse structurée et de préparation de refactorisation.
Elle peut inclure de très petites adaptations techniques si elles sont vraiment nécessaires pour clarifier la cartographie, mais elle ne doit pas lancer la migration complète vers la base.

Cette phase ne doit pas encore implémenter :
- la migration complète des paramètres en base
- le bootstrap de first-initialisation
- une UI d’administration
- des endpoints REST d’admin complets
- du RW sur `application.yml`

Périmètre attendu :

1. Recenser les paramètres existants
   Je veux identifier les propriétés actuellement dispersées :
- dans le YAML
- dans les `@ConfigurationProperties`
- dans les constantes du code
- dans les lectures directes ou usages implicites

Il faut repérer :
- les paramètres techniques
- les paramètres métier / administrables
- les cas hybrides
- les zones où la source de vérité est floue

2. Classer par nature et source cible
   Je veux distinguer clairement :
- configuration technique
- configuration administrable
- contenu éditable

Et pour chaque catégorie, déterminer la source cible la plus saine :
- YAML
- base de données
- fichiers dédiés

3. Identifier les accès à refactorer
   Je veux repérer les composants qui :
- lisent directement le YAML ou des properties devenues métier
- dépendent d’une source de config qui devra changer
- mélangent configuration statique et configuration administrable

L’objectif est de préparer un refactoring propre vers des services de lecture centralisés.

4. Produire une matrice de migration explicite
   Je veux un artefact clair, par exemple un fichier markdown de type :
- `configuration-inventory.md`
- ou `configuration-migration-plan.md`

Ce document doit contenir pour chaque paramètre ou famille de paramètres :
- nom / identification
- emplacement actuel
- catégorie (technique / administrable / contenu)
- source cible
- besoin de refactor
- remarque éventuelle
- besoin de bootstrap éventuel

5. Garder la phase structurée et réaliste
   Je veux une analyse ancrée dans le dépôt réel, pas un document théorique générique.
   Il faut partir de ce qui existe vraiment dans le code.

Attendus techniques :
- analyser réellement le dépôt
- identifier les classes/fichiers concernés
- produire l’artefact de cartographie/migration
- si utile, ajouter un minimum de structure de code ou de TODO clairs, mais sans lancer la migration complète
- garder le tout lisible, exploitable et maintenable

Ce que je veux explicitement éviter :
- migration prématurée
- document vague ou générique
- classification abstraite non reliée au code réel
- refactorisation large à ce stade
- sur-ingénierie

Livrables attendus :
1. artefact markdown de cartographie/migration
2. éventuelles petites adaptations de code strictement utiles
3. résumé court des constats principaux
4. liste des fichiers analysés/modifiés/créés
5. liste explicite des zones à refactorer dans la phase 18.2.A

Avant de coder :
- rappelle brièvement ton plan d’analyse en 5 à 10 étapes maximum
- puis réalise la phase
