Tu vas implémenter la phase 18.3 de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que les phases précédentes déjà réalisées, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur l’architecture déjà en place, notamment :
- runtime multi-agent
- policies
- tools
- mémoire
- bus inter-agent
- tasks
- traces
- rétention
- nouvelle stratégie de configuration (YAML = bootstrap/technique, DB = config administrable, fichiers = contenus métier)

Objectif de cette phase :
ne pas commencer par une grosse UI, mais stabiliser d’abord le modèle d’administration et exposer une administration technique simple, lisible et exploitable.

Cette phase ne doit pas encore implémenter :
- une vraie UI web riche
- un front d’administration complet
- des workflows complexes d’administration
- une gestion exhaustive de tous les cas métier
- une couche de présentation sophistiquée

Le but est de poser une base d’admin propre côté backend et exploitation.

Périmètre attendu :

1. Stabiliser d’abord le modèle d’administration
   Je veux consolider le modèle d’administration autour des briques déjà en place :
- agents
- policies
- tools
- mémoire
- bus
- tâches
- rétention

L’objectif est d’éviter de figer une UI sur un modèle encore confus ou mouvant.

2. Commencer par une admin technique simple
   Je veux une première couche d’administration légère, côté backend, par exemple via :
- properties propres
- endpoints REST ciblés
- états et traces lisibles

Attention :
- ne pas transformer cela en API géante d’admin
- exposer seulement des points utiles, cohérents avec l’état réel du projet
- rester orienté exploitation, debug et pilotage simple

3. Exposer des vues utiles et ciblées
   Je veux pouvoir commencer à consulter ou piloter techniquement des éléments pertinents, selon ce qui existe réellement déjà :
- état des agents
- tâches
- traces
- policies
- artefacts
- éventuellement configuration administrable

Attention :
- rester progressif
- ne pas prétendre tout administrer d’un coup
- garder les endpoints sobres et maintenables

4. Garder l’admin alignée sur le vrai modèle métier
   Je veux éviter :
- des endpoints bricolés pour une future UI hypothétique
- une couche d’admin qui contourne les services métier
- une exposition incohérente avec la source de vérité mise en place

L’admin doit s’appuyer sur le modèle existant, pas créer une seconde architecture parallèle.

5. Préparer la future UI sans la construire ici
   Cette phase doit préparer plus tard :
- une UI web d’administration
- des écrans de debug
- des vues de supervision
- des écrans de pilotage

Mais sans les implémenter maintenant.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/controllers/DTO nécessaires
- garder les responsabilités nettes entre services métier, endpoints d’admin et exposition technique

- garder le code lisible, testable et maintenable

- grosse API d’admin prématurée
- endpoints trop couplés à une UI non encore faite
- duplication de logique métier dans les controllers
- contournement des services existants
- sur-ingénierie
