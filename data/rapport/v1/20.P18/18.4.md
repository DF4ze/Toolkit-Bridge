Tu vas implémenter la phase 18.4 de mon projet “Toolkit-Bridge”.

Contexte :
Le cadre global du projet, la logique d’implémentation incrémentale, ainsi que la phase 18.3 déjà réalisée, te sont fournis séparément.
Tu dois t’y conformer strictement et t’appuyer sur le modèle d’administration et les endpoints/backend techniques déjà mis en place.

Objectif de cette phase :
préparer proprement une future UI web d’administration, sans construire dès maintenant une grosse interface, et en s’assurant que l’UI pourra se brancher sur un modèle déjà propre.

Cette phase ne doit pas encore implémenter :
- une UI web complète
- un front riche de production
- un dashboard final complet
- une administration exhaustive de toutes les briques
- des workflows d’édition complexes côté interface

Le but est de préparer la couche UI comme une couche de confort future, pas comme un pansement sur un backend mal modélisé.

Périmètre attendu :

1. Cibler les usages prioritaires
   Je veux clarifier les usages prioritaires d’une future UI d’administration, autour notamment de la capacité à :
- voir les agents
- voir les tâches
- voir les traces
- voir les policies
- voir les artefacts
- gérer les configurations liées
- aider au debug et à l’exploitation

L’objectif est de préparer une UI utile, pas décorative.

2. Préparer l’UI autour d’un modèle déjà propre
   Je veux que la future UI se branche sur les modèles, services et endpoints déjà stabilisés.
   L’interface doit devenir une couche de confort.
   Elle ne doit pas forcer des refontes importantes du backend.

3. Rendre l’exposition future plus lisible
   Je veux, si pertinent, clarifier ou consolider certains contrats d’exposition backend pour qu’ils soient plus adaptés à une future UI :
- DTO d’admin
- vues synthétiques
- regroupements cohérents
- états lisibles

Attention :
- ne pas construire ici toute l’UI
- ne pas sur-optimiser le backend pour un front hypothétique
- rester pragmatique

4. Préparer le debug et l’exploitation
   La future UI devra surtout aider à :
- comprendre l’état du système
- suivre l’exécution
- consulter la configuration administrable
- faciliter l’exploitation quotidienne

Je veux que cette logique soit visible dans les contrats exposés.

5. Réduire les futures refontes
   Je veux explicitement éviter qu’une future UI oblige à :
- refaire les modèles
- refaire les endpoints
- créer des vues incohérentes
- contourner les services métier

Cette phase doit donc consolider les points de branchement utiles.

Attendus techniques :
- créer ou adapter les classes/interfaces/services/controllers/DTO nécessaires si cela améliore réellement la préparation de la future UI
- garder les responsabilités nettes
- rester cohérent avec la phase 18.3 et les briques d’administration déjà stabilisées
- garder le code lisible, testable et maintenable
- ajouter ou adapter les tests pertinents si nécessaire

Ce que je veux explicitement éviter :
- implémentation prématurée d’un front complet
- backend sur-adapté à une UI encore absente
- endpoints artificiels ou trop spécifiques
- duplication entre exposition technique et exposition future UI
- sur-conception

Livrables attendus :
1. code complet de la phase 18.4
2. tests
3. résumé court des choix autour de la préparation de la future UI web d’administration
4. liste des fichiers modifiés/créés
5. points laissés ouverts pour les futures évolutions d’interface

Avant de coder :
- rappelle brièvement ton plan d’implémentation en 5 à 10 étapes maximum
- puis implémente
