# Lot 6 — Administration des LLM dans Marcel

Tu es un développeur senior Java Spring Boot.

Tu travailles sur mon projet **Tool-Bridge**, dont l’interface d’administration web s’appelle **Marcel**.

Les lots précédents ont déjà permis de mettre en place :
- le refacto des endpoints d’administration
- la sécurité simple par token/session
- le socle UI web Marcel
- le layout admin
- le dashboard
- les pages techniques en lecture seule

Je veux maintenant implémenter la première vraie section de **configuration métier** :
## l’administration des LLM

---

## Objectif de ce lot

Créer dans l’interface Marcel une section d’administration des LLM permettant au minimum :

- de lister les LLM configurés
- d’afficher leur détail
- de préparer ou permettre leur création / édition si cela est cohérent avec l’existant
- de rendre cette administration claire, exploitable et propre

⚠️ Important :
Je veux une implémentation **alignée sur le backend réellement disponible**.
Si le projet n’a pas encore un vrai modèle CRUD complet de LLM persistant, alors :
- ne pas inventer une usine à gaz
- ne pas simuler faussement un CRUD total
- mais poser une structure propre, exploitable et extensible

---

## Intention produit

Dans Marcel, un LLM correspond à une configuration exploitable par l’application, typiquement autour de notions comme :
- id / nom
- provider
- model
- endpoint/base URL si pertinent
- statut actif / enabled
- éventuellement capacités ou type de provider
- paramètres visibles utiles à l’admin

Je veux une UI d’administration claire et sobre, orientée exploitation/configuration, pas une page “debug brut”.

---

## Contraintes fortes

- Spring MVC + Thymeleaf
- Bootstrap 5
- jQuery léger uniquement si vraiment utile
- pas de SPA
- pas de React/Vue
- pas de build frontend Node
- pas de gros refacto backend hors sujet
- réutiliser au maximum l’existant
- éviter les appels REST internes
- privilégier les services/facades côté serveur
- ne pas exposer inutilement les secrets

---

# 1. Analyser l’existant LLM avant d’implémenter

Avant de coder, analyse le backend existant concernant les LLM :
- configuration actuelle
- services/facades existants
- endpoints REST disponibles
- représentation actuelle des providers / modèles / configs
- ce qui est réellement administrable aujourd’hui

## Important
Je veux que l’UI colle à la réalité du projet.

Donc :
- si aujourd’hui les LLM viennent principalement de la configuration applicative, ne fais pas semblant qu’ils sont pleinement CRUD en base si ce n’est pas vrai
- si un niveau d’édition minimal est possible, fais-le proprement
- sinon, mets d’abord en place une administration en lecture / consultation structurée, avec préparation propre pour l’édition future

---

# 2. Créer un controller MVC dédié aux pages LLM

Créer un contrôleur MVC dédié, par exemple :
- `LlmAdminPageController`
  ou nom équivalent cohérent avec le projet

## Routes attendues au minimum
- `GET /admin/llms`
- `GET /admin/llms/{llmId}`

## Si l’édition est réellement cohérente avec l’existant, ajouter aussi :
- `GET /admin/llms/new`
- `POST /admin/llms`
- `GET /admin/llms/{llmId}/edit`
- `POST /admin/llms/{llmId}`

Si cette partie n’est pas encore réaliste au vu du backend, ne force pas artificiellement un faux CRUD complet.
Dans ce cas :
- pose plutôt des placeholders propres
- ou limite ce lot à liste + détail + structure prête pour la suite

---

# 3. Réutiliser la couche métier existante

Je veux que tu réutilises l’existant autant que possible :
- facades
- services
- DTO
- configuration déjà disponible

Je ne veux pas de duplication de logique entre pages MVC et API REST.

Si l’existant ne fournit pas exactement le bon objet pour la vue, tu peux :
- créer un ViewModel dédié
- créer un mapper simple et propre
- mais sans sur-engineering

---

# 4. Créer la page liste des LLM

Créer une vue :
- `templates/admin/llms/list.html`
  ou équivalent propre

## Attendu
Afficher la liste des LLM configurés dans une table ou un ensemble de cards lisibles.

### Colonnes / infos utiles possibles
Selon les données réellement disponibles :
- identifiant
- nom
- provider
- modèle
- endpoint ou base URL si utile
- statut enabled/disabled
- type/capacités si disponibles

## UX attendue
- rendu lisible
- liens vers le détail
- état vide propre
- pas de surcharge visuelle
- cohérence avec le thème Marcel

---

# 5. Créer la page détail d’un LLM

Créer une vue :
- `templates/admin/llms/detail.html`

## Attendu
Afficher les informations détaillées d’un LLM sélectionné.

Je veux une présentation plus lisible qu’une simple table brute, par exemple :
- card de synthèse
- sections d’informations
- quelques badges/labels si utile

## Important
Ne jamais exposer de secret sensible en clair.
Si une clé/API key ou donnée sensible existe :
- la masquer
- ou ne pas l’afficher
- ou afficher seulement une version tronquée si c’est pertinent

---

# 6. Préparer ou implémenter l’édition si cohérent

Si le backend existant s’y prête vraiment, implémenter une édition simple et propre.

## Forme attendue
- page formulaire claire
- validation serveur
- messages de succès / erreur
- retour à la liste ou au détail après action

## Champs
Basés uniquement sur ce qui fait sens dans le projet actuel.

Exemples potentiels selon l’existant :
- name
- provider
- model
- baseUrl
- enabled

## Important
- ne pas inventer des champs non cohérents avec le projet
- ne pas prétendre persister quelque chose qui ne l’est pas réellement
- ne pas exposer/éditer les secrets de manière imprudente

Si l’édition n’est pas encore saine à ce stade :
- prépare simplement la structure
- laisse une UI de consultation propre
- et documente ce qui manque pour aller au CRUD complet
