# Lot 8 — Administration des Agents dans Marcel

Tu es un développeur senior Java Spring Boot.

Tu travailles sur mon projet **Tool-Bridge**, dont l’interface d’administration web s’appelle **Marcel**.

Les lots précédents ont déjà permis de mettre en place :
- le refacto des endpoints d’administration
- la sécurité simple par token/session
- le socle UI web Marcel
- le layout admin
- le dashboard
- les pages techniques en lecture seule
- la section d’administration des LLM
- la section d’administration des Bots Telegram

Je veux maintenant implémenter la section d’administration des **Agents**.

---

## Objectif de ce lot

Créer dans l’interface Marcel une section d’administration des Agents permettant au minimum :

- de lister les agents configurés
- d’afficher leur détail
- de visualiser clairement leur rattachement à un LLM et à un Bot Telegram
- de préparer ou permettre leur création / édition si cela est cohérent avec l’existant
- de rendre cette administration claire, exploitable et propre

⚠️ Important :
Je veux une implémentation **alignée sur le backend réellement disponible**.
Si le projet n’a pas encore un vrai modèle CRUD persistant complet pour les agents :
- ne pas inventer artificiellement une usine à gaz
- ne pas faire semblant qu’on a un CRUD complet si ce n’est pas vrai
- mais poser une structure propre, saine et extensible

---

## Intention produit

Dans Marcel, un Agent représente l’assemblage logique d’au moins :
- une identité d’agent
- un LLM
- un Bot Telegram
- éventuellement un type d’orchestrateur
- éventuellement un prompt système
- éventuellement un état enabled/disabled
- éventuellement d’autres paramètres d’exécution ou d’administration selon l’existant

L’UI doit rendre cela immédiatement compréhensible :
- quel agent utilise quel LLM
- quel agent est relié à quel bot Telegram
- quelle est sa configuration visible principale

Je veux une section admin claire, orientée configuration/exploitation, pas une page technique brute.

---

## Contraintes fortes

- Spring MVC + Thymeleaf
- Bootstrap 5
- jQuery léger uniquement si vraiment utile
- pas de SPA
- pas de React/Vue
- pas de build frontend Node
- pas de gros refacto backend hors sujet
- réutiliser au maximum l’existant
- éviter les appels REST internes
- privilégier les services/facades côté serveur
- ne pas exposer inutilement les secrets
- ne pas casser le couplage réel existant entre agents, LLM et bots

---

# 1. Analyser l’existant Agents avant d’implémenter

Avant de coder, analyse le backend existant concernant les agents :
- configuration actuelle
- propriétés Spring déjà présentes
- services/facades existants
- endpoints REST éventuellement disponibles
- représentation actuelle des agents
- lien avec les LLM
- lien avec les bots Telegram
- ce qui est réellement administrable aujourd’hui

## Important
Je veux que l’UI colle à la réalité du projet.

Donc :
- si aujourd’hui les agents viennent principalement de la configuration applicative, ne fais pas semblant qu’ils sont pleinement CRUD en base si ce n’est pas vrai
- si un niveau d’édition minimal est possible, fais-le proprement
- sinon, mets d’abord en place une administration en lecture / consultation structurée, avec préparation propre pour l’édition future

---

# 2. Créer un controller MVC dédié aux pages Agents

Créer un contrôleur MVC dédié, par exemple :
- `AgentAdminPageController`
  ou nom équivalent cohérent avec le projet

## Routes attendues au minimum
- `GET /admin/agents`
- `GET /admin/agents/{agentId}`

## Si l’édition est réellement cohérente avec l’existant, ajouter aussi :
- `GET /admin/agents/new`
- `POST /admin/agents`
- `GET /admin/agents/{agentId}/edit`
- `POST /admin/agents/{agentId}`

Si cette partie n’est pas encore réaliste au vu du backend, ne force pas artificiellement un faux CRUD complet.
Dans ce cas :
- pose plutôt des placeholders propres
- ou limite ce lot à liste + détail + structure prête pour la suite

---

# 3. Réutiliser la couche métier existante

Je veux que tu réutilises l’existant autant que possible :
- facades
- services
- DTO

Si l’existant ne fournit pas exactement le bon objet pour la vue, tu peux :

# 6. Préparer ou implémenter l’édition si cohérent

Si le backend existant s’y prête vraiment, implémenter une édition simple et propre.

## Forme attendue
- page formulaire claire
- validation serveur
- messages de succès / erreur
- retour à la liste ou au détail après action

## Champs
Basés uniquement sur ce qui fait sens dans le projet actuel.

Exemples potentiels selon l’existant :
- id
- name
- llmId / provider / model si applicable par référence
- telegramBotId
- orchestratorType
- enabled
- systemPrompt ou champ équivalent si cela existe réellement

## Important
- ne pas inventer des champs non cohérents avec le projet
- ne pas prétendre persister quelque chose qui ne l’est pas réellement
- ne pas casser la cohérence des relations existantes

## Très important
Si tu proposes un formulaire d’édition/création avec association à un LLM et à un Bot Telegram :
- réutiliser les données réellement disponibles
- proposer des listes déroulantes/select propres si possible
- éviter les saisies libres absurdes si les références connues existent déjà

Si l’édition n’est pas encore saine à ce stade :
- prépare simplement la structure
- laisse une UI de consultation propre
- et documente ce qui manque pour aller au CRUD complet
