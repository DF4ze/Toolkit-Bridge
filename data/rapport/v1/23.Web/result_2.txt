Après relecture :
Faiblesses / points discutables identifiés

[P2] Le contrôleur mélangeait routing MVC, composition de données dashboard, fallback technique et modèle de vue, ce qui augmentait le couplage et la dette de maintenance: AdminPageController.java.
[P2] Le web était couplé directement au DTO runtime/technique (TechnicalAdminView.Overview), ce qui brouillait la séparation UI vs runtime: AdminPageController.java.
[P3] Les fallback sur erreurs étaient silencieux (pas d’observabilité), donc diagnostic difficile en prod.
[P3] Les tests du contrôleur validaient surtout le routing, pas la robustesse de la composition dashboard.
Corrections proposées

Extraire la composition dashboard dans un service dédié.
Introduire un modèle de vue web dédié (ne plus exposer TechnicalAdminView dans le contrôleur).
Ajouter une journalisation explicite sur fallback.
Ajouter des tests unitaires métier du service dashboard (cas nominal + cas dégradé).
Corrections appliquées (pertinentes, sans ajout hors périmètre)
