Après relecture:
1. Faiblesses / points discutables (revue critique)
Exception trop générique dans la couche métier LLM
IllegalArgumentException mélangeait validation métier attendue et erreurs techniques potentielles.
Impact: gestion d’erreur MVC moins précise.
Réf: LlmAdminFacade.java:52

getLlm basé sur le DTO de liste
La recherche passait par listLlms() (DTO de vue), pas directement par le modèle config source.
Impact: couplage inutile entre lookup métier et projection UI.
Réf (avant correction): LlmAdminFacade.java:31

Tests de persistance pas assez précis
Les tests validaient juste “save appelé”, pas le contenu exact enregistré.
Impact: risque de régression silencieuse sur la structure persistée.

Dette restante (non bloquante, non corrigée ici)
Sauvegarde “liste complète” (saveOpenAiLikeProviders) => risque de lost update en cas d’écritures concurrentes.
Réf: AdministrableConfigurationGateway.java:63

Séparation config/runtime claire mais incomplète côté exploitation
C’est bien séparé, mais pas de mécanisme de reload runtime immédiat (message UI seulement).
C’est cohérent avec le périmètre, à traiter plus tard.

2. Corrections utiles proposées
Introduire une exception dédiée de validation admin LLM.
Découpler getLlm du DTO de liste.
Durcir les tests en vérifiant le payload exact persisté.

Exception métier dédiée

Ajout: LlmAdminValidationException.java
LlmAdminFacade lance désormais cette exception pour les erreurs de validation.
LlmAdminPageController capture explicitement cette exception.
Réf: LlmAdminFacade.java:52, LlmAdminPageController.java:92
Découplage modèle config / projection UI

getLlm lit directement OpenAiLikeProperties, puis mappe vers LlmAdminResponse.
Réf: LlmAdminFacade.java:31
Tests plus utiles

Vérification précise des listes persistées en create/update (pas juste “appel effectué”).
Réf: LlmAdminFacadeTest.java:75, LlmAdminFacadeTest.java:107
4. Hors périmètre (non ajouté volontairement)
Pas de nouveau comportement fonctionnel (pas de suppression, pas de reload runtime auto, pas de refacto infra store).
Pas d’extension vers orchestrator/memory/tooling/policy/workspace.
5. Résumé final propre
Séparation configuration/runtime: correcte pour ce lot (édition de config admin persistée, runtime non promis en hot-reload).
Qualité du modèle: améliorée (validation explicite, exception métier dédiée, lookup découplé du DTO UI).
Découplage inter-sous-systèmes: bon, aucun couplage ajouté avec orchestrator/memory/tooling/policy/workspace.
Lisibilité et noms: cohérents, controller MVC fin, façade claire.
Tests: plus robustes sur les cas critiques de persistance.
Dette restante principale: écriture “full list” non transactionnelle fine en concurrence, à traiter dans un lot d’industrialisation config.
