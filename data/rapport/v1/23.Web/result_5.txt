Après relecture:
Faiblesses / points discutables (revue critique)

[P2] Couplage involontaire au mécanisme global de flash messages
Le controller utilisait infoMessage dans le modèle pour un message statique de page.
Risque: collision avec les vrais flash infos des actions futures.
Réf: TelegramBotAdminPageController.java

[P3] Duplication mineure de setup navigation
activeNav / activeTelegramBotNav étaient répétés sur chaque handler.
Réf: TelegramBotAdminPageController.java:24

[P3] Dette résiduelle acceptable
Le modèle Telegram admin reste read-only car la source est toolkit.telegram.supervision (configuration runtime), pas une config administrable persistée type LLM.
C’est un choix cohérent, mais il faudra une couche admin persistante dédiée pour passer au CRUD.

Corrections utiles proposées

Retirer l’usage de infoMessage dans le controller Telegram (préserver le canal flash pour les vraies actions).
Afficher l’information de limite actuelle directement dans les templates Telegram (alert locale).
Factoriser la navigation dans une méthode privée du controller.
